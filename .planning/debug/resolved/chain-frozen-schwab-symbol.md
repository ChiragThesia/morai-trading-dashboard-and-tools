## Root Cause (pre-diagnosed — three compounding bugs)

This session was opened AFTER a full read-only diagnosis. Root cause is confirmed; the work is the TDD fix. Do NOT re-investigate auth — it is ruled out (see Eliminated).

**BUG 1 — malformed chain symbol (the root).** `apps/sidecar/chain_proxy.py:199` calls `client.get_option_chain("SPX")` and `("SPXW")`. Schwab `/marketdata/v1/chains` accepts ONLY `$SPX` (a single `$SPX` call returns BOTH SPX and SPXW contracts). Proven live: `symbol=$SPX`→200; `SPX`/`SPXW`/`$SPXW`→`400 "Check Param Values"`. Unbounded `$SPX` (no scoping) times out → the fix must pass `$SPX` PLUS scoping params (`strike_count`/`range`/`from_date`/`to_date`) as `packages/adapters/src/schwab/market/chain-adapter.ts` already does, and collapse the two SPX+SPXW calls into one `$SPX` call. Verify the schwab-py `get_option_chain` kwarg names/enums for those params.

**BUG 2 — 400→AUTH_EXPIRED mislabel.** `apps/sidecar/chain_proxy.py:210` maps EVERY non-200 → `503 {"error":"AUTH_EXPIRED"}`. A `400` bad-param must not be reported as auth expiry. Distinguish 400/param-error (→ a param/fetch error) from a real 401 (→ AUTH_EXPIRED). This is what made the outage masquerade as a dead token for weeks.

**BUG 3 — CBOE fallback keyed on the wrong signal.** `apps/worker/src/main.ts:155` `selectChainSource` picks per-run by market TOKEN FRESHNESS only (`fresh`/`stale`→sidecar/Schwab, `AUTH_EXPIRED`/`none_yet`→CBOE). It never falls back to CBOE on an actual Schwab CALL failure, so a healthy token + broken Schwab request = permanent failure, CBOE safety net silent. Add call-failure → CBOE fallback (so a Schwab fetch that errs at runtime retries via CBOE within the same invocation).

Priority: **B1 revives the pipeline; B2+B3 stop recurrence.**

## Resolution

root_cause: Three compounding bugs. (1) apps/sidecar/chain_proxy.py passed the `root` query param straight through as the literal Schwab symbol ("SPX"/"SPXW"); Schwab's /marketdata/v1/chains only accepts "$SPX", so every request 400'd — the schwab_chain source never wrote a row, ever. Reopened after live deploy: fixing the symbol alone was not sufficient — the unbounded strike ladder over a 90-day window still overflowed Schwab's own API gateway (502 "Body buffer overflow"), so a `strike_count` bound was also required. (2) chain_proxy.py mapped every non-200 status (including that 400) to the same 503 {"error":"AUTH_EXPIRED"} as a real 401, so the param bug was misreported as a dead token for weeks even though the token refreshed cleanly. (3) selectChainSource picked schwabFetchChain purely from token freshness at the start of the call and never reacted to the call's actual outcome, so once a token was fresh/stale, any live Schwab call failure (for any reason) permanently blacked out the pipeline with the CBOE safety net never engaging.

fix:
- BUG 1 (original): chain_proxy.py always requests symbol "$SPX" (ignoring the `root` param value for the Schwab call itself), scoped with from_date=today/to_date=today+90d.
- BUG 1 (reopened, session 2): added `strike_count=50` (schwab-py kwarg -> Schwab's `strikeCount` query param) to the same get_option_chain() call — date-bounding alone still overflowed Schwab's gateway at prod scale (live-verified 502); strike_count=50 brings it under the limit (live-verified 200/4.08MB). Value matches the existing convention in packages/adapters/src/schwab/market/chain-adapter.ts. Deleted the disproven "from_date/to_date alone is enough" code comment.
- BUG 2: chain_proxy.py only maps status_code==401 to {"error":"AUTH_EXPIRED"}; any other non-200 now returns {"error": f"SCHWAB_FETCH_ERROR_{status}"}.
- BUG 3: selectChainSource's fresh/stale branch now returns a composed fetcher that tries schwabFetchChain first and falls back to cboeFetchChain within the same call on any failure, instead of returning schwabFetchChain directly.

verification: Self-verified via TDD (RED confirmed for the right reason, then GREEN) for all four fixes (BUG 1 original + BUG 1 strike_count + BUG 2 + BUG 3), plus full regression suites: apps/sidecar 69/69 passed (both BUG 1 sessions), packages/core 617/617 passed, `bun run typecheck` clean, `bun run lint` clean (one pre-existing unrelated boundaries-plugin warning). B2 + B3 additionally live-verified in prod (session 2, see Current Focus LIVE STATUS). BUG 1's strike_count fix is NOT yet verified live — constraints forbid live Schwab/CBOE calls or prod queries in this session, and the fix has not been deployed. Awaiting operator sidecar redeploy + human confirmation that the next live RTH `fetch-schwab-chain` run writes `source='schwab_chain'` rows.
