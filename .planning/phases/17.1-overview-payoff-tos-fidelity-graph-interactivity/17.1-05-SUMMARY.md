# Phase 17.1 Plan 05: Positions-Box Expiry/DTE Reformat Summary

**The docked positions table's compact `Nd → Nd` DTE cell is now a two-line `{expiry(s)} / {DTE(s) + calendar width}` cell under an `Expiry / DTE` header, driven by a new pure, unit-tested `formatExpiryCell` helper.**

## Performance

- **Duration:** ~12 min
- **Completed:** 2026-07-04T02:22:10Z
- **Tasks:** 2 completed
- **Files modified:** 2

## Accomplishments

- Added `formatExpiryCell(input)` to `Overview.tsx`: a pure, exported helper taking a discriminated-union input (`{kind:"calendar", frontOccSymbol, backOccSymbol, dteFront, dteBack}` or `{kind:"single", occSymbol, dte}`). For a calendar it parses both leg OCC symbols via `parseOccSymbol`, formats each expiry with the existing `gexAsOf` short-month convention (`toLocaleString("en-US", {month:"short", day:"numeric"})`), and builds `line1 = "{frontExpiry} → {backExpiry}"` / `line2 = "{dteFront}d/{dteBack}d · {dteBack−dteFront}d wide"`. For a single leg it's `line1 = "{expiry}"` / `line2 = "{dte}d"`. A `parseOccSymbol` failure falls back to `line1: "—"` without throwing — no `!`, no `as`.
- Replaced `Row.dte: string` with `Row.expiry: ExpiryCell` (`{line1, line2}`), populated in `buildRows` for both calendar rows (via `CalendarGroup.dteFront/dteBack` — no re-parse of the DTE, just the two expiry dates) and single-leg rows (via the pre-existing DTE computation, now defaulting to `0` instead of short-circuiting the whole cell on a parse failure).
- Reformatted the DTE `<td>` as a `flex-col` stack: line1 at `text-[11px] text-muted-foreground`, line2 at `text-[9px] text-dim` — both existing Phase-9 sizes/tokens, no new ones introduced.
- Renamed the `COLS` header entry `"DTE"` → `"Expiry / DTE"`, keeping the existing right-alignment.
- Confirmed `BookSummary` (which reads only `Row.legs`) compiles unaffected, and no other reader of the removed `Row.dte` field remains.

## Decisions Made

- `formatExpiryCell` takes a discriminated-union input rather than a `CalendarGroup`/`BrokerPositionResponse` directly, keeping it decoupled from `pair-calendars.ts` types and trivially unit-testable with literal values, matching the plan's own Task 1 acceptance criteria (literal `dteFront:32`/`dteBack:59`).
- On a single-leg occSymbol parse failure, the DTE passed into `formatExpiryCell` now defaults to `0` (previously the whole DTE cell short-circuited to a bare `"—"`) — this mirrors `pair-calendars.ts`'s own `dte()` helper, which already defaults calendar-leg DTEs to `0` on a parse failure. No existing test asserted the old all-dash behavior, and unparseable `occSymbol`s do not occur in the current live book — not a functional regression.
- Split the two tasks strictly at the plan's boundary: Task 1 landed `formatExpiryCell` as pure and unwired (verified `apps/web`'s own `tsc --noEmit` clean in that intermediate state); Task 2 did the `Row`/`buildRows`/render wiring. See "Issues Encountered" for how this was caught and corrected mid-execution.
