## Summary

This phase is a **surgical restyle + wiring fix**, not new architecture: every data path already exists (`repriceScenario`, `daysForward`, `pairPositionsIntoCalendars`, `PositionsTable`'s `excluded` Set). The work is (1) lifting the `excluded` Set one level up so it can feed both `calendarPositions[].included` (chart) and the table total (today it only reaches the total), (2) adding a native `<input type="date">` + step arrows that compute a `daysForward` integer and feed it into the existing `ScenarioParams`, (3) recoloring two `LinePath` strokes and fixing the y-domain/x-tick math in `PayoffChart.tsx`, and (4) reformatting one table cell using data (`dteFront`/`dteBack`) that already exists on `CalendarGroup` plus a `parseOccSymbol` call for the two expiry dates that is not yet made in `buildRows`.

The two ⚑ RESEARCH REQUIRED flags could not be fully resolved to HIGH confidence. **D-03a (colors):** Schwab's own documentation and multiple secondary sources consistently describe the *default* Risk Profile scheme as a **purple "today" curve and a blue "at-expiration" curve** (green instead of blue on the light/SSE scheme) — not magenta/cyan. The user's own reference screenshots (which would settle this) are not present on disk; only `mockups/tos-reference/README.md` exists. Because D-03 is an explicit, locked user instruction naming magenta/cyan specifically, and TOS lets users customize per-curve colors (the user may run a customized or newer TOS color scheme not covered by the docs consulted), the locked magenta (`#e91e8c`) / cyan (`#22d3ee`) values from the UI-SPEC should stand as **the executable target**, but this is flagged `[ASSUMED]`/pending a real pixel-check the moment the screenshots are re-dropped — do not treat "purple/blue" as a silent correction. **D-02 (checkbox coupling):** no Schwab/TOS documentation (official manual, Schwab Learn articles, or TickerTape explainers) describes a chart-only vs. chart+aggregate distinction for the Positions-and-Simulated-Trades checkbox column — the graph IS the aggregate of every checked row in that single panel, which is consistent with (but does not conclusively prove) the "unified" interpretation. The UI-SPEC's D-02a "unified" assumption is the most defensible reading and matches the in-repo Analyzer pattern exactly; treat it as confirmed absent contrary evidence, not as independently verified TOS behavior.

**Primary recommendation:** Implement OVW-06 by lifting `excluded` state from `PositionsTable` to `Overview`, threading it into `buildCalendarPosition`'s `included` field (replacing the `Overview.tsx:144` hardcode) and into `PositionsTable` as a prop+callback; implement OVW-05 with a timezone-safe local-date day-diff helper (own pure function, unit-tested) rather than raw millisecond subtraction; implement OVW-04 as three isolated changes to `PayoffChart.tsx` (color constants, `computeYDomain` combining both curves, a new `buildXTicks(X_MIN, X_MAX)` pure function); implement OVW-03 by adding an expiry-date lookup via `parseOccSymbol(c.front.occSymbol)`/`parseOccSymbol(c.back.occSymbol)` inside `buildRows`.

### Pattern 1: State lifting for a single source of truth (OVW-06)

**What:** `PositionsTable`'s `excluded` Set (`Overview.tsx:295`, `useState<ReadonlySet<string>>(new Set())`) currently lives entirely inside the child component. `Overview`'s `calendarPositions` (built at `Overview.tsx:744-747`) is computed independently, with `included: true` hardcoded in `buildCalendarPosition` (`Overview.tsx:144`). To unify, the `excluded` Set must move up to `Overview` (a plain `useState` next to `hoveredRowKey`/`selectedRowKey`), and be passed down to `PositionsTable` as `{ excluded, onToggle }` props instead of being locally owned.

**When to use:** Any time two sibling/parent-child pieces of UI need to observe and mutate the same state — the React idiom is "lift state to the nearest common ancestor," which here is `Overview` itself (the chart and the table are siblings under it).

**Example (signature-level, not full implementation):**
```typescript
// Source: React state-lifting idiom, applied over existing Overview.tsx/PositionsTable signatures
```

### Pattern 2: Timezone-safe date-picker day-math (OVW-05)

**What:** `<input type="date">`'s value is a plain `"YYYY-MM-DD"` string. `new Date("YYYY-MM-DD")` parses it as **UTC midnight**, not local midnight — the same class of UTC/local-timezone bug this project has hit twice before (CBOE timestamps, per project memory). Comparing that UTC-midnight Date against `new Date()` (local, current time-of-day) via raw millisecond subtraction can push the computed `daysForward` off by one depending on the user's UTC offset and time of day.

**When to use:** Any time a native date input's string value must become an integer "days from today" count.

**Example:**
```typescript
// Source: MDN <input type="date"> value format + Date constructor semantics
// Parse the input's "YYYY-MM-DD" as LOCAL calendar date components, not via new Date(string):
function parseLocalDateInput(value: string): Date {
  const [y, m, d] = value.split("-").map(Number);
  return new Date(y, m - 1, d); // local-timezone constructor, NOT UTC-parsing string constructor
}

function localMidnight(d: Date): Date {
  return new Date(d.getFullYear(), d.getMonth(), d.getDate());
}

function daysBetween(from: Date, to: Date): number {
  const MS_PER_DAY = 86_400_000;
  return Math.round((localMidnight(to).getTime() - localMidnight(from).getTime()) / MS_PER_DAY);
}
```

### Pattern 3: Combined-curve y-domain (OVW-04)

**What:** `computeYDomain` (`PayoffChart.tsx:146`) currently derives `{lo, hi}` from `baseExpirationCurve` alone. Per UI-SPEC, it must combine **both** the today/date curve and the @exp curve so a near-flat T+0 line is never squashed against a tall @exp tent.

**Example:**
```typescript
// Source: PayoffChart.tsx:146 (existing function, ported to take two curves)
function computeYDomain(
  curveA: ReadonlyArray<PayoffPoint>,
  curveB: ReadonlyArray<PayoffPoint>,
): { lo: number; hi: number } {
  const all = [...curveA, ...curveB];
  if (all.length === 0) return { lo: -500, hi: 500 };
  let lo = Infinity, hi = -Infinity;
  for (const p of all) {
    if (p.pl < lo) lo = p.pl;
    if (p.pl > hi) hi = p.pl;
  }
  if (lo === hi) { lo -= 100; hi += 100; }
  const pad = (hi - lo) * 0.18;
  lo -= pad; hi += pad;
  if (lo > 0) lo = 0;
  if (hi < 0) hi = 0;
  return { lo, hi };
}
// Caller (PayoffChart component body) changes from
//   computeYDomain(baseExpirationCurve)
// to
//   computeYDomain(todayCurve, baseExpirationCurve)
```

**Important integration detail:** `positionSetSignature` (built at `Overview.tsx:764-766`) drives *when* `PayoffChart` recomputes the y-domain (`PayoffChart.tsx:214-219`, keyed off this signature). It currently includes only `p.id` + IV-status, **not** inclusion state. Once OVW-06 wires `excluded` into `included`, toggling a calendar off changes the curves but will **not** trigger a y-domain recompute unless the signature also encodes exclusion state (e.g. append `:${p.included}` per position). Missing this is a silent staleness bug, not a crash — easy to overlook in review.

### Pattern 4: Round-number x-axis ticks (OVW-04)

**What:** `PayoffChart.tsx:385` hardcodes `[7000, 7200, 7400, 7600, 7800]` as a separate literal from the `X_MIN`/`X_MAX` domain constants (`6900`/`7900`, `PayoffChart.tsx:99-100`). UI-SPEC requires these be *computed* from the domain rather than duplicated as an independent literal, so they can never drift out of sync if the domain constants ever change.

**Example:**
```typescript
// Source: derived generically from PayoffChart.tsx's existing X_MIN/X_MAX constants
function buildXTicks(min: number, max: number, targetCount = 5): ReadonlyArray<number> {
  const rawStep = (max - min) / (targetCount - 1);
  const roundSteps = [25, 50, 100, 200, 250, 500, 1000];
  const step = roundSteps.find((s) => s >= rawStep) ?? roundSteps[roundSteps.length - 1];
  const first = Math.ceil(min / step) * step;
  const ticks: number[] = [];
  for (let v = first; v <= max; v += step) ticks.push(v);
  return ticks;
}
// For X_MIN=6900, X_MAX=7900 this reproduces [7000,7200,...,7800] when step snaps to 200 —
// same visual result as today, but now derived, not duplicated.
```

### Anti-Patterns to Avoid

- **Two independent `excluded`/inclusion states (one in `PositionsTable`, one implied by the hardcoded `included: true` in `Overview`):** this is the literal bug OVW-06 is fixing. Do not "fix" it by adding a second synchronized Set — lift to one.
- **Using `Date` millisecond subtraction directly on `<input type="date">` values without normalizing to local midnight:** reproduces the CBOE-UTC bug class this project has already been burned by twice (see project memory `morai-phase2-production-lessons`).
- **Re-deriving the `@exp` curve from the picked date:** `bookPLAtExpiry` must stay parameterless w.r.t. `daysForward` — it is locked to `pos.frontDte` by design (D-01). Do not thread `daysForward` into the expiration curve's pricing call.
- **Hand-rolling a calendar popover for the date picker:** violates the zero-new-deps lock; native `<input type="date">` already provides `min`/`max` clamping and dark-mode theming (`colorScheme: "dark"`) for free.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Date picker UI + native OS calendar popup | Custom `<Popover>` + calendar grid component | `<input type="date">` with `style={{ colorScheme: "dark" }}` | Zero-dep, gets `min`/`max` range clamping, keyboard input, and dark-mode rendering natively across Chrome/Safari/Firefox; UI-SPEC already locked this choice |
| Local-date parsing from a date string | Ad-hoc string-splitting scattered across call sites | One small pure `parseLocalDateInput`/`daysBetween` helper (Pattern 2 above), unit-tested once | Timezone bugs are exactly the class of defect that has recurred in this project (CBOE, per memory); a single tested helper prevents re-deriving the fix incorrectly at each call site |
| Round-number axis tick generation | A second hardcoded tick array "for the new range" | `buildXTicks(min, max)` pure function (Pattern 4) | The whole point of the fix is to stop hardcoding literals that can drift from the domain constants |

**Key insight:** every "don't hand-roll" item in this phase is really the same warning — this is a restyle/wiring phase over an engine that already works; the risk is re-introducing bespoke one-off logic (a second exclusion Set, a second date-math implementation, a second tick array) instead of routing through the one correct path that already exists or is being added once.

## Common Pitfalls

### Pitfall 1: `<input type="date">` UTC-vs-local day-count drift
**What goes wrong:** `daysForward` computed one day off (or, near midnight in some timezones, two days off) from what the user picked.
**Why it happens:** `new Date("YYYY-MM-DD")` parses as UTC midnight; subtracting from `new Date()` (local "now," with a nonzero time-of-day component) mixes UTC-anchored and local-anchored instants.
**How to avoid:** Use Pattern 2's `parseLocalDateInput` + `localMidnight` + `daysBetween` helpers; never call `new Date(dateInputValue)` directly, and never diff against a `Date` that still carries the current time-of-day.
**Warning signs:** the picked date and the chart's projected date visually disagree by exactly one day, or the disagreement only appears for some users/timezones and not others.

### Pitfall 2: y-domain staleness when a calendar is toggled off
**What goes wrong:** After unchecking a calendar row, the chart's curves change but the y-axis scale doesn't rescale, so the (now smaller) P&L range looks squashed or the axis has dead space.
**Why it happens:** `positionSetSignature` (the memo key gating y-domain recompute in `PayoffChart.tsx`) doesn't currently encode inclusion state — only position id + IV status.
**How to avoid:** Extend the signature string built in `Overview.tsx:764-766` to include each position's `included` flag.
**Warning signs:** toggling a checkbox visibly changes curve shape but the y-axis labels stay pixel-identical.

### Pitfall 3: Reintroducing the OVW-06 decoupling bug via a second state copy
**What goes wrong:** A "quick fix" adds an `excluded` prop to `PositionsTable` while leaving its own internal `useState` in place (syncing the two with a `useEffect`), instead of removing the internal state entirely.
**Why it happens:** Minimal-diff instinct — touching the internal `useState` line feels riskier than adding a new prop alongside it.
**How to avoid:** Delete `PositionsTable`'s internal `excluded` `useState` entirely; it must become a controlled component for this one piece of state.
**Warning signs:** Two `useState<ReadonlySet<string>>` calls anywhere in the diff; a `useEffect` that copies one Set into another.

### Pitfall 4: `computeYDomain`/`findZeroCrossings` re-export drift
**What goes wrong:** `PayoffChart.tsx:816` re-exports `computeYDomain` for its existing unit tests (`PayoffChart.test.tsx`). Changing its signature (single curve → two curves) without updating the existing test call sites causes a type error that TDD's RED step should catch — but only if the existing tests are run, not skipped.
**Why it happens:** Signature changes on a re-exported pure function are easy to make without grepping all call sites first.
**How to avoid:** `rg computeYDomain` across `apps/web/src` before changing the signature; update every call site (component body + existing tests) in the same commit.
**Warning signs:** `tsc --noEmit` failure on the test file after the signature change (should be caught immediately since this is `apps/web`, which has its own `typecheck` script).

### Pitfall 5: TOS color/behavior claims presented as verified when they are not
**What goes wrong:** Treating `#e91e8c`/`#22d3ee` or the "unified checkbox" behavior as confirmed TOS fact rather than a locked-but-unverified user instruction, and skipping the planner-mandated human pixel-check.
**Why it happens:** The UI-SPEC already made a confident-sounding "best-effort match" call, which reads as settled.
**How to avoid:** Keep the UI-SPEC's own explicit caveat intact through planning and execution — a `checkpoint:human-verify` (or equivalent UAT step) against the re-dropped `mockups/tos-reference/*.png` screenshots must remain in the plan; do not silently drop it because the colors "look plausible."
