## Typography

Exactly **4 concrete font sizes** are in use across this surface (Phase-9 inherited scale —
this phase reuses them verbatim and introduces none). Every new element (date picker, step
arrows, reformatted expiry cell) maps onto one of these four:

| Role | Size | Weight | Line Height | Font | New elements that use it |
|------|------|--------|-------------|------|--------------------------|
| Micro (label / badge / axis tick / section label / expiry line 2) | 9px | 400 (section label 600) | default | JetBrains Mono; section label Space Grotesk `tracking-[0.09em] uppercase` | `Date:` label, ‹/› arrows, `Today` button, `32d/59d · 27d wide` line, `Expiry / DTE` header |
| Table / row data | 11px | 400 | default | JetBrains Mono, `tabular-nums` | positions-box expiry line 1 (`Aug 8 → Sep 5`), date-input value |
| Body / data base | 12px | 400 | 1.45 | JetBrains Mono (`body` default) | inherited baseline (no new use) |
| Emphasis value | 14px | 700 | default | Space Grotesk | chart tooltip P&L readout (unchanged) |

Notes on the merge (to satisfy the ≤4-size rule without inventing values):
- The 9px "micro" bucket absorbs the former 9px and 9-10px micro-label rows. Section labels
  ship at the 10px top of this micro tier (`text-[10px]` in `system/index.tsx`) and the axis
  tick / smallest data glyphs at 9px — all treated as one micro tier, not separate scale steps.
- The "emphasis value" is fixed at a single concrete **14px** (was a 13-15px range: the chart
  tooltip's 13px SPX label and 15px P&L value both round to this one declared bucket) — no new
  size is introduced by this phase; the tooltip is unchanged Phase-17 code.

No new weights: the app uses exactly two families — 400 (regular, JetBrains Mono default) and
600/700 (semibold/bold, Space Grotesk headings and emphasis values). Do not introduce a third
weight for the date picker or reformatted DTE cell.

## Phase-Specific Component Contracts

Supplementary detail beyond the 6 standard dimensions, because this phase is chart-heavy and
the planner/executor need an unambiguous rendering contract.

### Date picker + step arrows (OVW-05)

- **Placement**: a new row in the payoff-hero `Panel`, between the existing header row
  (`SectionLabel` + GEX/mark freshness badges) and the existing curve-legend row (T+0/@exp/γ
  flip/walls swatches). Left-aligned: `Date: [‹] [date input] [›] [Today]`.
- **Behavior**: picking a date (or stepping ‹/›) computes `daysForward` = whole days between
  today and the picked date, and feeds it into the existing `ScenarioParams.daysForward` for
  the "today"/date curve only (`scenario.payoffCurve`) — **the `@exp` curve
  (`scenario.expirationCurve`) is NOT re-derived from the picked date; it stays fixed** at
  each position's front expiry (locked in D-01, matches TOS's model where the expiration
  profile never moves).
- **Range clamp (D-01a)**: `min` = today, `max` = the earliest front expiry across included
  calendars (a calendar projected past its own front expiry is degenerate). When there are no
  included calendars, `max` = today (picker effectively disabled, matches empty-state).
- **Step increment**: 1 day per ‹/› click (Claude's Discretion, confirmed in CONTEXT.md).
- **Default**: today (`daysForward: 0`), matching the shipped Phase 17 baseline — no visual
  change when the picker is untouched.
- **Reuse note**: this ports `Analyzer.tsx`'s `daysForward` slider pattern (already wired
  end-to-end through `repriceScenario`) — the Overview version replaces the slider control
  with the TOS-style picker + arrows, not a new pricing path.

### TOS-fidelity payoff chart (OVW-04)

- **Curve colors**: see Color section override above (magenta today/date, cyan @exp).
- **Y-axis auto-scale fix**: `computeYDomain` currently derives the y-domain from
  `baseExpirationCurve` alone (`PayoffChart.tsx:146`), which is the literal cause of "@exp
  dwarfing a near-flat T+0" named in OVW-04's success criteria. The y-domain must be computed
  from **both** curves' combined min/max (today/date-projected curve AND @exp curve) so a
  near-flat T+0 line is never squashed to a hairline against a tall @exp tent.
- **X-axis ticks**: replace the current hardcoded `[7000, 7200, 7400, 7600, 7800]` strike
  label array (`PayoffChart.tsx:385`) with **evenly-spaced round-number ticks** computed from
  the live `X_MIN`/`X_MAX` domain (e.g. 5 ticks at a round step — 100 or 200 points depending
  on spot level) — this is the "mixed round/key-level x-ticks" problem named in OVW-04. GEX
  wall/flip vertical reference lines (put wall, call wall, γ flip) stay as separate overlay
  lines with their own labels — they are not part of the axis tick row itself.
- **Everything else unchanged**: the 9-layer z-order, gridlines, profit-zone fill, GEX wall
  lines, spot marker, crosshair/tooltip — none of that is in scope for the TOS restyle; only
  the two curve colors, the y-domain calc, and the x-tick generation change.

### Per-calendar series selection (OVW-06)

- **No new visual component.** The existing positions-table row checkbox
  (`PositionsTable`'s `excluded` Set, `Overview.tsx:295`) is the single control.
- **The wiring gap being fixed**: today `calendarPositions` for `repriceScenario` are built
  with `included: true` hard-coded (`Overview.tsx:144`), so unchecking a row currently affects
  only the P&L total/greeks, never the chart. This phase routes the same `excluded` Set into
  the chart's `calendarPositions` inclusion flag — unchecking a row must remove that
  calendar's contribution from **both** curves (unified behavior, D-02a working assumption,
  pending the TOS-behavior research note in CONTEXT.md D-02).
- **Visual feedback**: unchanged — the row keeps its existing `opacity-40` dim when excluded
  (`Overview.tsx:364`). No new highlight/dim treatment needed on the chart curves themselves;
  they simply stop including that calendar's contribution once its checkbox is unchecked.

### Positions-box expiry/DTE reformat (OVW-03)

- **Column**: rename header `DTE` → `Expiry / DTE`, keep `text-right` alignment.
- **Cell layout**: two stacked lines inside the same `<td>` (flex-col), replacing the current
  single-line `{frontDte}d → {backDte}d` string built in `buildRows()`
  (`Overview.tsx:239-256`).
  - Calendar row: `Aug 8 → Sep 5` (11px, `text-muted-foreground`) then `32d/59d · 27d wide`
    (9px, `text-dim`) — short-month date format matches the existing `gexAsOf`/`markAsOf`
    convention already in `Overview.tsx` (`toLocaleString("en-US", { month: "short", day:
    "numeric" })`).
  - Single-leg row: `Aug 8` then `32d` — no "wide" second metric.
- **Days-between math**: `backDte − frontDte`, computed from the same `CalendarGroup.dteFront`
  / `dteBack` fields already returned by `pairPositionsIntoCalendars` — no new data source.
