# Phase 17.1 Plan 02: PayoffChart TOS-Fidelity Restyle (Axis, Ticks, Color Seam, WR-03) Summary

**PayoffChart's y-axis now scales from both P&L curves combined, x-ticks are derived from a round-number generator instead of a hardcoded array, curve colors are prop-configurable with brand-matching defaults (Analyzer untouched), and the WR-03 setState-in-render antipattern is cleared — all without changing a single pixel of the currently-shipped Analyzer chart.**

## Accomplishments

- `computeYDomain` now takes `(todayCurve, expCurve)` and scans both for min/max, so a near-flat T+0 curve is never squashed by a tall @exp tent; the `{lo:-500,hi:500}` fallback (both curves empty) is preserved. Confirmed by grepping every call site (`Pitfall 4`) — the only callers were inside `PayoffChart.tsx` itself (two `useMemo`/`useEffect` blocks) plus the test file, all updated in the same commit.
- New pure `buildXTicks(min, max, targetCount=5)` replaces the hardcoded `[7000,7200,7400,7600,7800]` literal with a round-step-snapped derivation from `X_MIN`/`X_MAX`; reproduces the exact same five ticks today, but can no longer silently drift from the domain constants.
- WR-03 (17-REVIEW.md, `PayoffChart.tsx:214-227` pre-fix): the y-domain signature-lock and the `fitY`/`onFitYConsumed` refit were both running as side effects inside `useMemo` render-phase bodies. Moved both to `useEffect`. A harness test reproducing the exact antipattern (a parent component's `setState` invoked as `onFitYConsumed` during `PayoffChart`'s render) confirmed the pre-fix code triggers React's "Cannot update a component while rendering a different component" warning, and the post-fix code does not.
- `todayCurveColor`/`expirationCurveColor` optional props added to `PayoffChartProps`, defaulting to the existing `VIOLET`/`GRAY_MUTED` constants. Threaded into: the T+0 curve stroke (net-book, layer 2/9), the T+0 BE dashed line + label (layer 8), the @exp curve stroke (layer 4), and the @exp BE dashed line + label (layer 7). Gridlines, zero line, GEX wall lines, spot marker, profit-zone fill, glow filter, and the highlighted single-position overlay curves are unchanged — verified by an explicit test that passing the new color props does NOT change `highlighted-t0-curve`/`highlighted-exp-curve` strokes.
- `--color-tos-magenta: #e91e8c` added to `index.css`'s `@theme` block, beside the existing tokens; `--color-cyan` (already present) reused, not re-added.
- Because the new props' defaults equal today's constants, `Analyzer.tsx` (which passes neither prop) is provably unaffected — verified by the full `Analyzer.test.tsx` suite staying green (6/6) throughout all three tasks.

## Decisions Made

- `buildXTicks`'s step-selection formula uses `rawStep = (max - min) / targetCount`, diverging from `17.1-RESEARCH.md`'s Pattern 4 example (`/(targetCount - 1)`), which — for this file's own `X_MIN=6900`/`X_MAX=7900` domain — snaps to step 250 (`[7000,7250,7500,7750]`) rather than the plan's explicitly stated acceptance criterion of step 200 (`[7000,7200,7400,7600,7800]`, matching today's hardcoded array). The plan's `<behavior>`/`<acceptance_criteria>` blocks are the authoritative spec; the research pattern's own worked example was internally inconsistent with its stated output. Verified via `buildXTicks(6900, 7900)` unit test matching the exact literal set.
- Followed project `.claude/rules/tdd.md` ("commit at green only") over the generic executor RED/GREEN-commit-split convention, consistent with 17.1-01's established pattern: RED was run and confirmed (assertion/type-mismatch failures, not import errors) before each implementation, but only the green state was committed per task.
