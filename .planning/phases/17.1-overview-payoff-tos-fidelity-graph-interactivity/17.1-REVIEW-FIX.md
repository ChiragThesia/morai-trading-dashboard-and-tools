### WR-01: Date-projection bound over-constrained by IV-non-convergent calendars

**Files modified:** `apps/web/src/screens/Overview.tsx`
**Commit:** a44ef24
**Applied fix:** Extended the `bounds` `useMemo` filter so `computeProjectionBounds` receives
only calendars that both are `included` AND have `frontIvStatus !== "non-convergent"` and
`backIvStatus !== "non-convergent"` — matching the engine's `includedForT0` inclusion rule so a
non-convergent calendar (which draws no curve) no longer caps `maxDaysForward`. Applied exactly as
the review suggested. `Overview.test.tsx` 33/33 green.

### WR-02: Kernel-parity test never actually compares `computePositionGreeks` output

**Files modified:** `apps/web/src/lib/scenario-engine.test.ts`
**Commit:** a1d130e
**Applied fix:** Replaced the two vacuous assertions (`directGreeks.delta` compared to itself,
`typeof pgResult === "object"`) with a genuine parity check. The reviewer's literal suggestion
(`toBeCloseTo(directGreeks.delta * netQty)`) would have FAILED: `computePositionGreeks` derives T
from the OCC expiry (`parseOccSymbol` + `Date.now()`, 365.25-day year), not from `backDte/365` as
`directGreeks` does — those T values differ materially (OCC expiry 2026-08-08 vs the fixture's 69d).
So the fix reprices a fresh `bsmGreeks` call with the SAME expiry-derived T, strike, and type the
helper uses, then asserts `pgResult.value.greeks.{delta,gamma,theta,vega}` equal that kernel output
scaled by `netQty` (and `netQty === 1`), guarded by `expect(pgResult.ok/parsed.ok).toBe(true)`.
Added a `parseOccSymbol` import from `@morai/shared`. This now genuinely fails if the helper's
kernel wiring regresses. `scenario-engine.test.ts` 17/17 green.

### WR-03: Crosshair tooltip positioned with logical viewBox coordinates as CSS pixels

**Files modified:** `apps/web/src/components/charts/PayoffChart.tsx`
**Commit:** ae51d09
**Applied fix:** Added `renderedWidth` to the `crosshair` state and populate it from the already-
computed `svgRect.width` in `handlePointerMove`. The tooltip `left` now maps the viewBox-unit
`crosshair.x` into rendered pixels via `(crosshair.x / SVG_W) * crosshair.renderedWidth + 14`,
clamped to `crosshair.renderedWidth - 180`. This is derived from `crosshair.x` (definitively viewBox
units, as the reviewer confirmed the SVG line consumes it correctly), so it is correct regardless of
`localPoint`'s coordinate-space quirks. `PayoffChart.test.tsx` 18/18 green.
**Note — requires human visual verification:** no automated test asserts the tooltip's pixel
position; the mapping is correct by construction but should be eyeballed against the crosshair line
at a few viewport widths.
