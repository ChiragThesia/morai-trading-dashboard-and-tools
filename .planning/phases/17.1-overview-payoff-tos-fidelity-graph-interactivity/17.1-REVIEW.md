# Phase 17.1: Code Review Report

**Reviewed:** 2026-07-03
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

Reviewed the Phase 17.1 decimal-polish work: the timezone-safe `date-projection` helper lib,
the TOS-fidelity `PayoffChart` restyle (y-domain / x-ticks / curve-color seams), and the
Overview date-picker + per-calendar inclusion toggle + expiry/DTE cell.

The core date math is sound — `daysBetween` normalizes both endpoints to local midnight and
rounds, so DST and negative-UTC-offset drift (the documented CBOE bug class) are handled, and
the fast-check round-trip locks it. The CR-01 regression-sensitive path (non-convergent
calendars excluded from T+0/@exp) is correctly wired through `includedForT0`/`includedForExpiry`
and has a dedicated regression test.

No blockers found. The most material finding is a **date-projection bound that is over-constrained
by IV-non-convergent calendars** — a position that contributes nothing to the drawn curves still
caps the picker's reachable date range (WR-01). The other warnings are a misleading/vacuous parity
assertion (WR-02, notable given this project's documented history of green suites masking prod
bugs), a tooltip positioned in logical viewBox units treated as CSS pixels (WR-03), and a
mixed-horizon hover overlay after date projection (WR-04).

## Warnings

### WR-01: Date-projection bound over-constrained by IV-non-convergent calendars

**File:** `apps/web/src/screens/Overview.tsx:852-859` (and `apps/web/src/lib/date-projection.ts:65-74`)
**Issue:** `bounds` is derived from `calendarPositions.filter((p) => p.included).map((p) => p.frontDte)`,
i.e. every checkbox-included calendar. But a calendar whose front/back leg IV did not converge is
excluded from BOTH payoff curves by the scenario engine (`includedForT0`/`includedForExpiry`), so it
draws nothing — yet its `frontDte` still feeds `Math.min(...includedFrontDtes)` and therefore caps
`maxDaysForward` / the input `max` attribute. Example: one convergent calendar (frontDte 40) + one
non-convergent (frontDte 5) → the picker clamps projection to +5d even though the only curve on the
chart is projectable to +40d. The projection horizon is governed by a position the user cannot see.
**Fix:** Exclude non-convergent legs from the bound the same way the curves do:
```ts
const bounds = useMemo(
  () =>
    computeProjectionBounds(
      calendarPositions
        .filter(
          (p) =>
            p.included &&
            p.frontIvStatus !== "non-convergent" &&
            p.backIvStatus !== "non-convergent",
        )
        .map((p) => p.frontDte),
      today,
    ),
  [calendarPositions, today],
);
```
(Consider exporting the engine's `includedForT0` predicate so the Overview and the engine share one
inclusion rule rather than re-deriving it.)

### WR-02: Kernel-parity test never actually compares `computePositionGreeks` output

**File:** `apps/web/src/lib/scenario-engine.test.ts:154-168`
**Issue:** The test is titled "per-position greeks equal a direct bsmGreeks call AND the Plan-06
computePositionGreeks output" and the comment claims "We verify the kernel itself is the same by
comparing directGreeks." The `computePositionGreeks` half of that contract is asserted only by
`expect(directGreeks.delta).toBeCloseTo(directGreeks.delta, 10)` (a value compared to itself, always
passes) and `expect(typeof pgResult).toBe("object")` (true for any Result). `pgResult` is never
compared to `bsmGreeks` or to the scenario greeks. A regression in `computePositionGreeks`' kernel
usage would sail through this test despite its name. Given this codebase's documented history of green
suites hiding production bugs, a parity check that only appears to verify parity is a real reliability
defect.
**Fix:** Assert the actual value, e.g. `if (pgResult.ok) expect(pgResult.value.greeks.delta).toBeCloseTo(directGreeks.delta * netQty, 4)`
with a real qty, or delete the vacuous lines and the "verify the kernel" comment so the test's claimed
coverage matches what it checks.

### WR-03: Crosshair tooltip positioned with logical viewBox coordinates as CSS pixels

**File:** `apps/web/src/components/charts/PayoffChart.tsx:736-754` (value set at 314-339)
**Issue:** `crosshair.x` is computed as `logicalX = point.x * (SVG_W / svgRect.width)` — i.e. viewBox
units in `[0, 1000]`. The SVG line consumes it correctly (drawn inside the viewBox). But the HTML
tooltip does `left: Math.min(crosshair.x + 14, SVG_W - 180)`, applying a logical-unit value (max 820)
as CSS pixels on an absolutely-positioned div whose real pixel width is the rendered SVG width, not
1000 (the chart is `width: 100%` in a responsive `1fr` column, so typically ~1000–1140px, and less on
narrow viewports). The tooltip therefore drifts from the crosshair line it annotates (and its `SVG_W - 180`
clamp is in the wrong unit space). The numbers shown are correct; only the placement is off.
**Fix:** Convert back to pixel space for the tooltip, e.g. capture `svgRect.width` into crosshair state
and use `left: Math.min(crosshair.pxX + 14, svgRect.width - 180)`, or store `point.x` (already CSS px)
alongside the logical value and position the tooltip from the pixel value. (Note: appears to predate
17.1 but lives in a reviewed file and is a genuine rendering defect.)

### WR-04: Highlighted single-position overlay stays at daysForward:0 while the net book projects forward

**File:** `apps/web/src/screens/Overview.tsx:913-923` (consumed at 1066-1068)
**Issue:** `highlightedScenario` is always repriced with `daysForward: 0`, but the net-book `todayCurve`
passed to the chart reflects the date-picker's `daysForward`. When a user projects the book forward
(e.g. +10d) and then hovers/selects a row, the dimmed net-book T+0 curve is drawn at +10d while the
highlighted single-position T+0 curve is drawn at 0d. The two violet/magenta curves are then at
different time horizons, so the highlighted position looks richer purely because of the horizon
mismatch — a misleading comparison on exactly the interaction this phase added.
**Fix:** Thread the same `daysForward` into the highlight reprice:
```ts
const params: ScenarioParams = { spot, daysForward, ivShift: 0, rate: DEFAULT_RATE, divYield: DEFAULT_DIV };
```
and add `daysForward` to the `useMemo` dependency array.

### IN-04: `as unknown as` double cast + eslint-disable violates the no-`as` rule

**File:** `apps/web/src/screens/Overview.test.tsx:96-97`
**Issue:** `mockUsePositions.mockReturnValue({ ... } as unknown as ReturnType<typeof usePositions>)` with
a suppressing `eslint-disable`. The project's typescript rule bans `as` assertions outright; a typed
mock factory or a Zod-parsed fixture would satisfy it. Test-only and acknowledged, but it is the exact
pattern the rule forbids.
**Fix:** Build the mock return via a helper typed to `ReturnType<typeof usePositions>` (or `satisfies`)
rather than a double cast.

### IN-05: `parseLocalDateInput` accepts out-of-range fields instead of rejecting them

**File:** `apps/web/src/lib/date-projection.ts:27-39`
**Issue:** The doc says malformed input returns `null` and "never Invalid Date," but `"2026-99-99"` and
`"0000-00-00"` pass the `Number.isFinite` checks and produce a rolled-over local `Date` (e.g. month 99
overflows into a later year) rather than `null`. Impact is contained because `resolveDaysForward` clamps
the result, and a native `<input type="date">` won't emit such values — but the behavior contradicts the
docstring and the test only asserts finiteness, not `null`, for these inputs.
**Fix:** Range-check the parsed fields (`month` 1–12, `day` 1–31) and return `null` on out-of-range, or
soften the docstring to match actual behavior.

### IN-06: fast-check property describes a symmetry it never asserts

**File:** `apps/web/src/lib/scenario-engine.test.ts:227-265`
**Issue:** The comment block details a "symmetry property: P&L at +spot_offset and -spot_offset ...
magnitude ratio should be reasonable (< 20)," but the body only checks that heatmap cells are finite and
that a near-strike cell's `|pl| < 100_000`. No symmetric pair is ever compared, so the named property is
not actually under test.
**Fix:** Either implement the +/- offset magnitude-ratio assertion, or trim the comment to the invariant
actually checked (finiteness + bounded near-strike P&L).

### IN-07: Hardcoded chart X-domain vs. Overview's 5800 spot fallback renders the spot marker off-canvas

**File:** `apps/web/src/components/charts/PayoffChart.tsx:113-114` (with `apps/web/src/screens/Overview.tsx:793`)
**Issue:** `X_MIN`/`X_MAX` are fixed at 6900–7900. Overview falls back to `spot = gex?.spot ?? 5800`
when GEX is unavailable, so `spotX = xScale(5800)` is negative and the spot line/dot (and the nearest-point
readout) render outside the plot area. Only reachable in the no-GEX empty state, so low impact, but the
hardcoded domain will silently misplace the marker any time spot leaves 6900–7900.
**Fix:** Derive the X-domain from the curve data (or clamp/guard the spot marker), or align the Overview
fallback spot with the chart domain.
