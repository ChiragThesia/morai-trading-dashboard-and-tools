# Phase 17.1 Plan 01: Timezone-Safe Date-Projection Helpers Summary

**Zero-dependency date-projection library (`toDateInputValue`, `parseLocalDateInput`, `daysBetween`, `resolveDaysForward`, `computeProjectionBounds`) that turns an `<input type="date">` value into a clamped, NaN-safe `daysForward` integer, with a fast-check round-trip property killing the UTC-vs-local day-math bug class before any UI wiring consumes it.**

## Accomplishments
- `parseLocalDateInput`/`toDateInputValue`/`daysBetween` implemented with local-timezone-only `Date` construction (`new Date(y, m-1, d)`), never the UTC-parsing single-string constructor — killing RESEARCH Pitfall 1 (the CBOE-UTC bug class this project has hit twice, inverted direction).
- fast-check round-trip property (1000 runs) proves `daysBetween(today, parseLocalDateInput(toDateInputValue(today+offset))) === offset` for any offset in [0, 365] and any local time-of-day, independent of the runner's timezone.
- `resolveDaysForward` composes the Task 1 helpers into the single entry point the Overview date-picker will call: empty/malformed value → `0` (never `NaN`), out-of-range dates clamp into `[0, maxDaysForward]`.
- `computeProjectionBounds` derives the native `<input>` `min`/`max` attributes and `maxDaysForward` from the included front-expiry DTEs (`Math.min(...includedFrontDtes)`), so a calendar can never be projected past its own front expiry (D-01a).
