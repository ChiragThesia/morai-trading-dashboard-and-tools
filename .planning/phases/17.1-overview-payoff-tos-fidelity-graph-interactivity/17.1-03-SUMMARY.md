**Lifted `excludedCalendars` state from PositionsTable to Overview so one Set now drives both the payoff-chart curves and the table total — closing the OVW-06 decoupling bug where unchecking a row moved the P&L total but left the chart curve unchanged.**

## Accomplishments
- `buildCalendarPosition`'s hardcoded `included: true` replaced with a real 5th param sourced from the lifted `excludedCalendars` Set — the literal bug (row checkbox reaching only the P&L total) is closed.
- `PositionsTable` is now a fully controlled component for inclusion: no internal `useState`, no local `toggle`, checkbox `onChange` calls the `onToggleExcluded` prop directly. Exactly one `useState<ReadonlySet<string>>` exists in the whole file.
- `positionSetSignature` appends each position's `included` flag, so `PayoffChart`'s y-domain memo key changes on toggle (RESEARCH Pitfall 2 — no silent staleness).
- Checkbox `aria-label` updated to `Include {label} in risk profile & total` per the UI-SPEC copywriting contract.
- Proved the CR-01 leg-IV-non-convergence exclusion (`includedForT0 = pos.included && !isIvExcludedFromT0`) composes correctly with the new lifted-state wiring — a non-convergent calendar stays off both curves even with its checkbox left checked.

## Decisions Made
- **Resequenced Task 1 to include the full controlled-component wiring.** The plan's Task 1 action explicitly states the RED test should fail "because inclusion is hardcoded" when the checkbox is toggled — that framing only holds if the checkbox already reaches `buildCalendarPosition`'s `included` param. Deferring the checkbox wiring to Task 2 (as literally described in Task 2's own action block) would have made Task 1's RED test fail for the wrong reason (missing wiring, not the hardcoded value). I completed the state lift AND the controlled-component wiring in Task 1, then used Task 2's commit for its remaining unique scope: the `aria-label` copywriting-contract text change and explicit regression tests proving "exactly one Set, no syncing effect, one click updates both surfaces." Verified via `bunx vitest run` (RED confirmed failing for the hardcoded-`included` reason, then GREEN) and via the acceptance criteria's own grep checks (`useState<ReadonlySet<string>>` count == 1, no `included: true` match, no `useEffect` in the file) — all pass. This is a Rule 1/Rule 3 style task-boundary adjustment, not a scope change: every line the plan specified for Tasks 1+2 combined was written; only which commit it landed in shifted.
- `excludedCalendars` (plan-specified name) holds both calendar-row keys (`$SYM|strike|type`) and single-leg-row keys (raw `occSymbol`) since `PositionsTable` renders both row kinds from one Set — no collision risk (the two key formats never overlap), and `buildCalendarPosition` only ever looks up by calendar key, so single-leg exclusions correctly affect only the table total (per plan: "Single-leg rows keep affecting only the table total... calendar rows now drive both").
