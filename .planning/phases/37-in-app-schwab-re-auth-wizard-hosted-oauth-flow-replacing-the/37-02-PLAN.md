<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server (core+adapter) → sidecar | The adapter is the only TS code that calls the privileged sidecar admin endpoints; it must carry the shared secret and must not leak upstream error detail. |
| sidecar response → browser (via contracts) | The exchange response schema is the last type-level gate preventing a code/state echo from ever reaching the SPA. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-37-02 | Information Disclosure | reauth-adapter.ts failure paths | high | mitigate | Log only `error.constructor.name` on every failure path — never the message, redirect URL, or code (positions-reconciler.ts convention); adapter returns Result, never throws upstream detail outward. |
| T-37-03 | Elevation of Privilege | reauth-adapter.ts request | high | mitigate | Every request attaches the injected `X-Sidecar-Admin-Token` header — the adapter is how the server authenticates to the otherwise-unauthenticated sidecar admin surface (paired with 37-04's header guard). |
| T-37-06 | Information Disclosure | reauth Zod contracts | medium | mitigate | All four schemas are `.strict()` and the exchange response is a bare `{ app, ok }` — no code/state/error-detail field can be added or parsed through, so the browser never receives leakable server detail. |
</threat_model>
