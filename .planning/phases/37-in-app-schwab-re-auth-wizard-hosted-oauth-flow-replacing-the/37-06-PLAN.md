<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Schwab redirect → SPA (morai.wtf landing) | The `?code=&state=` in the URL is sensitive; it must be stripped before render and never logged, though it is unavoidably visible to the hosting access log. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-37-02 | Information Disclosure | web callback handling | high | mitigate | Capture + `history.replaceState` strip at module scope before any render; no component ever reads window.location for the code; the code/redirect URL is never logged or rendered in any copy/tooltip/error (UI-SPEC content constraint). |
| T-37-07 | Information Disclosure | auth code in Vercel access logs | low | accept | Standard public-redirect OAuth posture (CONTEXT-accepted): the code is single-use, ~30s TTL, useless without the app secret; our own stack never logs it. Documented, not mitigated in code. |
| T-37-08 | Tampering | auto-resume exchange | low | accept | No confirm screen and no await-able hop between landing and the exchange POST (UI-SPEC-locked) keeps the round trip inside Schwab's ~30s code TTL; a StrictMode second effect can't re-fire because the URL is already stripped and the captured href is consumed once. |
</threat_model>
