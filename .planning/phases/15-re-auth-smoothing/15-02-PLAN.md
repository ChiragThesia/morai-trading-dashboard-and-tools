<objective>
Harden the existing operator re-auth CLI (apps/sidecar/seed_token.py) and write the operator runbook (AUTH-06). The re-auth mechanism is ~80% built: the two-step (authurl -> exchange) and one-shot (login) flows already dual-write broker_tokens and reset the refresh_issued_at 7-day anchor. This plan commits the pending hardening diff, corrects the post-exchange restart instruction, and captures the whole flow as a runbook a future operator (or Claude session) can execute.

Purpose: SC2 — the operator can run a local re-auth flow (manual-flow -> token_write -> Postgres) that restores Schwab auth without shipping new code.
Output: committed + corrected seed_token.py; new docs/operations/schwab-reauth-runbook.md; TOPIC-MAP Operations entry; one-line reconciliation of a stale deployment.md claim.

CRITICAL PROVISIONAL DECISION — surface at plan review (RESEARCH Summary + Open Question 3; orchestrator provisional decision 2):
- AUTH-06's roadmap wording "without a Railway redeploy ... sidecar picks up the new token on its next auto-refresh cycle" is VERIFIED INACCURATE. The running sidecar reads token_read_func() exactly once at client construction (schwab-py auth.py:540) and never re-reads Postgres on refresh cycles. A freshly-written token row is picked up ONLY by restarting the process. Resolution: read "no redeploy" as "no code rebuild/ship" and satisfy it with railway redeploy --service sidecar -y (restarts the existing image, no rebuild — verified in Railway CLI v4.11.0). We deliberately do NOT add a sidecar reload endpoint or file-watcher (D-02 minimal-attack-surface; that adds code to the auth-critical path). The user may override this at review.
- AUTH-05's status surface is UNAFFECTED by this finding: readTokenFreshness reads Postgres timestamps directly, so /api/status shows restored freshness immediately after the write, before any restart.

PROVISIONAL DECISIONS (user AFK — override at review): D-02 (CLI two-step, no web form), D-02a (both apps one pass), D-02b (runbook in docs/).
</objective>
