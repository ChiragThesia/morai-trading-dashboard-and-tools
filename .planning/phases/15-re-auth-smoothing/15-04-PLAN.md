<objective>
Add the AUTH-05 warning-log side effect (D-03): a single warning log line fires when the refresh token first crosses the T-24h threshold. Per architecture-boundaries.md §2, packages/core stays I/O-free — the boolean/seconds are computed in core (toAppTokenStatus, 15-01), and the console.warn lives in a driving adapter. To satisfy "a single warning log line" without duplicating logic across the HTTP route and the MCP tool, the warning is a decorator around the shared getStatus use-case, wired once in the composition root so BOTH surfaces inherit it.

Purpose: SC1 — "the status surface logs a warning; the alert fires before the token expires."
Output: a `withRefreshExpiryWarning` adapter decorator (in-process per-app latch) + its test + one-line composition-root wiring.

PROVISIONAL DECISION — surface at plan review (RESEARCH Open Question 1; orchestrator provisional decision 3):
- Dedup semantics: D-03 says "a single warning log line fires when freshness computation FIRST crosses T-24h." We implement a cheap in-process per-app latch: log on the null->non-null transition, re-arm on non-null->null (so a re-auth followed by a later re-approach warns again). This matches "first crosses" precisely and avoids spamming a line every 30s poll. Alternative readings (fire-on-every-poll, or once-per-process-lifetime) are noted for the user to confirm. A process restart resets the latch — acceptable (re-arming a warning is safer than staying silent).
- D-01: the warning is a passive log, consistent with "no push adapter."
</objective>
