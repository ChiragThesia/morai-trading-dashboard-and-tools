## Phase Goal

**As a** trader using Claude Code, **I want to** pull my Schwab trade transactions for any date range through the `get_transactions` MCP tool over the shared contract, **so that** I can review trade history in-session without leaving my agent.

**Mode:** mvp

This plan delivers SLICE 1 of Phase 7: the `get_transactions` MCP tool as a *verified, contract-locked* capability. The tool's production code already exists (built in Phase 4 / BRK-02: `registerGetTransactionsTool` in `tools.ts`, registered in `server.ts`, `getTransactions` wired in `main.ts`). What is missing — and what SC1 + every BRK-03 acceptance criterion actually demands — is **behavioral test coverage and the documentation entry**. This slice closes that gap. After this plan, a real user (Claude Code) can call `get_transactions` and we can *prove* it returns a contract-valid payload, defaults to 90 days, and pauses (never throws) on AUTH_EXPIRED.

<objective>
Lock and verify the existing `get_transactions` MCP tool against BRK-03: document the tool in the MCP architecture doc (docs-first), then add behavioral tests that prove the SPEC acceptance criteria — msw-style faked use-case returns a `transactionsResponse`-valid payload for a range, omitted params default to last-90d, AUTH_EXPIRED yields the typed `brokerageAuthExpiredPayload` (no throw), and the contract is the single shared `@morai/contracts` source (MCP-02, enforced at compile time).

Purpose: BRK-03's acceptance is test-shaped, not build-shaped — the tool is built but untested and undocumented. This slice makes the capability trustworthy.
Output: `docs/architecture/mcp-and-plugins.md` tool entry; `apps/server/src/adapters/mcp/get-transactions.test.ts`.
</objective>
