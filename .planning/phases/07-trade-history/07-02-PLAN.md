## Phase Goal

**As a** trader who just connected Schwab, **I want to** backfill my historical trade transactions over an arbitrary past date range with one command, **so that** my older trades flow into fills and my calendar journal reflects real history — not just the rolling 7-day window.

**Mode:** mvp

This plan delivers SLICE 2 of Phase 7: a historical backfill CLI for `sync-transactions`. Today the worker only ever syncs a rolling 7-day window (`txWindowFrom`/`txWindowTo` in `apps/worker/src/main.ts`). There is no way to pull history. This slice adds a pure date-window chunker in core (the unit under fast-check), then a thin CLI driving adapter that runs the existing `sync-transactions` use-case per chunk over a caller-supplied `[from,to]`, chunked within Schwab's lookback cap, idempotent on re-run. Built and tested entirely OFFLINE (in-memory fills twin + faked transactions fetch); a live run additionally needs valid Schwab tokens (operator prerequisite, out of code scope).

<objective>
Add a historical trade-history backfill: a PURE `chunkDateRange(from, to, maxDays)` domain function in `packages/core` (hexagon — imports only `@morai/shared`, returns `Result`, errors when the range exceeds the lookback cap), proven by fast-check (no gaps, no overlap-dupes, each window ≤ maxDays, cap respected); then a thin backfill CLI in `apps/worker` that runs the existing `sync-transactions` use-case once per chunk over `[from,to]`, writing `fills` idempotently (deterministic fill ids), plus a `bun run` script. Document the resolved Schwab lookback cap and the new CLI (docs-first).

Purpose: BRK-04 — pull arbitrary trade history into fills → calendar events, chunked and idempotent, without live auth to build/test.
Output: `chunkDateRange.ts` (+ example + property tests), `backfill-transactions.ts` (+ test), worker bun script, jobs.md + TOPIC-MAP doc updates.
</objective>
