# Phase 19 — UI Design Contract

> Visual and interaction contract for PICK-01/02/03: **NOT a redesign.** Phase 18 already shipped
> the full picker layout (`Analyzer.tsx`, `CandidateCard.tsx`, `WhyPanel.tsx`, `TermStructureChart.tsx`,
> `EntryExitPlan.tsx`, `PayoffChart.tsx`) against a frozen fixture. Phase 19 replaces the fixture
> with live data (PICK-02: "no layout change") and adds exactly the visual surface required by two
> additive contract needs: **(a)** chain-snapshot staleness + source must be visible on every
> candidate-score surface (success criterion 3, D-15/D-16), and **(b)** degraded-context honesty
> tags for GEX/events (D-17) plus new loading/error/empty states the synchronous fixture never
> needed (D-18/D-19). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Scope Statement (read first)

Per the phase brief: **PICK-02 requires "no layout change"** when `Analyzer.tsx` swaps its
`pickerSnapshotFixture` import for a live `usePicker()` fetch. This UI-SPEC therefore does **not**
redefine the 3-column grid, card anatomy, breakdown bars, why-panel, term-structure chart, or
entry/exit plan — those contracts are FROZEN from `18-UI-SPEC.md` and remain unchanged. This spec
covers only the following additive surfaces:

1. **Per-card staleness + source badge** (D-16/D-15) — new sub-line tag on `CandidateCard`.
2. **Context-status indicators** (D-17) — new inline tags for `gexContextStatus`/`eventsContextStatus`
   when either is `stale`/`missing`.
3. **Live-data states** (D-18/D-19) — loading skeleton, fetch-error+retry, and two distinct empty
   messages (cold-start vs. zero-candidates-passed-filter) on the `CandidateRail`.
4. **Guard-case rendering** — already fully specified in `18-UI-SPEC.md`'s "Null-FwdIV / guard-case
   render branch" table; unchanged, restated here only as a pointer (no new decision).

If a plan or executor finds itself redesigning card anatomy, panel layout, or column widths to
accommodate this phase's needs, that is out of contract — the additive tags below MUST fit inside
the existing sub-line row and panel shells without changing their dimensions.

## Visuals

**No new focal point.** The ranked candidate-cards rail (left) and payoff-center (center) remain
the two-anchor screen from Phase 18. The only visual additions are:

- A **staleness+source tag** appended to each `CandidateCard`'s existing sub-line row (same row as
  the `DTE … · debit … · θ … · vega …` text and the event tags `{ev}◂f`/`{ev}◂b`/`clean` — the new
  tag is one more inline `<span>` in that same flex-wrapped line, not a new row).
- A **context-status tag** (amber, `.tag`-style, matching the existing `{ev}◂f` visual grammar)
  appended to the same sub-line, shown ONLY when `gexContextStatus !== "ok"` or
  `eventsContextStatus !== "ok"` for that snapshot (snapshot-level fields, so every card in a given
  fetch shows the same context tags — D-17 mirrors the fwdIv-guard precedent of tagging at the
  affected term, never hiding it).
- **Rail-level states** replacing the candidate list: a loading skeleton (mirrors `MacroCard`'s
  "Loading …" text pattern, no shadcn `Skeleton` pulse — see Copywriting Contract), an error card
  with retry, and the cold-start vs. zero-passed-filter empty messages (D-18) — all rendered
  **inside the existing `Panel`/`PanelHeading` "Suggested calendars" shell** from
  `CandidateRail` (18-UI-SPEC), never a new panel or route.

No changes to: 3-column grid, column widths, card border/selected-state treatment, breakdown bar
rendering, payoff-center overlay, why-panel, term-structure chart, or entry/exit plan.

## Phase-Specific Component Contracts

### Per-card staleness+source tag (D-15/D-16, PICK-02 success criterion 3)

- **Placement**: appended to `CandidateCard`'s sub-line row (`apps/web/src/components/picker/CandidateCard.tsx`,
  the `<div className="mt-0.5 font-mono text-[9px] text-dim">` block), after the existing event
  tags — same flex-wrapped line, no new row, no card-height change beyond natural text wrap.
- **Anatomy**: `<span className="ml-1 flex items-center gap-1 rounded-sm bg-raise px-1 py-0.5">` containing
  a `size-1.5 rounded-full` dot (`bg-up` if fresh, `bg-amber` if stale — reuse `GEX_FRESH_MS`-style
  threshold from `Market.tsx`, exact threshold value is planner's call) + text `as of {HH:MM} · {source}`.
- **Data source**: `pickerSnapshotResponse.asOf` + `pickerSnapshotResponse.source` (D-15) — snapshot-level
  fields repeated per-card (D-16: "every card is self-contained"), not per-candidate fields. Every
  `CandidateCard` in a given fetch renders the identical tag content.
- **Guard**: never renders a blank/NaN age — if `asOf` fails to parse, render `as of —` (em-dash),
  never a JS error or `Invalid Date` string.

### Context-status tags (D-17)

- **Placement**: same sub-line row, after the staleness+source tag. Rendered conditionally — only
  present when the corresponding status is not `"ok"`.
- **Anatomy**: identical `<span>` tag styling to event tags (`rounded-sm bg-raise px-1 py-0.5
  text-amber`), text `GEX unavailable` / `events unavailable`. Both may appear simultaneously
  (space-separated, same row, wraps naturally).
- **Scoring linkage**: when a context tag is shown, the corresponding breakdown bar (`gexFit` for
  GEX, `eventAdjustment` for events) MUST show its guard/zero-contribution render — reuse the exact
  guard-bar visual already defined in `18-UI-SPEC.md` for the `fwdEdge` guard case (bar width 0,
  track-color-only, caption `n/a`) rather than inventing a new zero-state visual for these two bars.

### Rail live-data states (D-18/D-19)

- **Loading**: replaces the candidate list only (Panel/PanelHeading/legend text stay mounted).
  Centered `Loading candidates…` text, `data-testid="picker-loading"`. Shown when
  `usePicker().isPending && data === undefined` (mirrors `MacroCard`'s exact condition).
- **Error**: replaces the candidate list. `Couldn't load candidates.` (text-down) + `Retry` mini-button
  that calls the query's `refetch()`. `data-testid="picker-error"`.
- **Cold-start empty**: `data === undefined` after settling (no `picker_snapshot` row exists yet) →
  `picker-empty-cold-start` copy (see Copywriting Contract).
- **Zero-candidates empty**: `data.candidates.length === 0` (snapshot exists, engine filtered
  everything via net-θ>0) → `picker-empty-filtered` copy, interpolating `data.asOf`.
- **State precedence** (mutually exclusive, checked in this order): loading → error → cold-start-empty
  → zero-candidates-empty → populated rail. Never show two of these simultaneously.

### Guard-case rendering (fwdIV null)

Unchanged — see `18-UI-SPEC.md` "Null-FwdIV / guard-case render branch" table. This phase makes the
guard case reachable with real (not fixture) data; no new visual decision required.
