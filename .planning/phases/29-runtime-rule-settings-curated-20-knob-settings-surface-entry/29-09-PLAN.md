<objective>
Build the thin cross-cutting settings bounded context: storage-facing ports plus the two
use-cases that back the GET/PUT surface. Per RESEARCH.md, this context owns ONLY JSONB
storage orchestration + the display merge — it NEVER imports picker/exits/analytics `domain/`
code (architecture rule 7). The full default knob object is INJECTED by the composition root
(29-13) as a plain value computed from the three engines' resolve functions, so this context
stays engine-agnostic.

Reset-per-group semantics (orchestrator resolution of RESEARCH Open Question 2): a PUT with
`{ picker: null }` deletes that group's stored override keys. `setRuleOverrides` translates the
request's nullable groups into the clean stored partial (no nulls persisted).

Purpose: the use-cases + ports the server surface (29-13) and the storage repo (29-08) wire to.
Output: `ForReadingRuleOverrides`/`ForWritingRuleOverrides` ports, `makeGetRuleSettingsUseCase`,
`makeSetRuleOverridesUseCase`, and pure `mergeStoredOverrides`/`computeEffective` helpers.
</objective>
