<objective>
Wire the resolved regime config into the regime board — the simplest of the three engines
(RESEARCH Pitfall 3): `getRegimeBoard` computes live per HTTP/MCP request with no worker job and
no persisted snapshot, so a threshold override takes effect on the very next read. Add
`readRuleOverrides` to `GetRegimeBoardDeps`, read it fresh inside the use-case body, resolve
`resolveRegimeRuleConfig(overrides.regime)`, and pass each `{ warn, crisis }` pair into the
matching `bandX` call.

Threading map (call sites in getRegimeBoard.ts):
- `bandVixTermStructure(value, config.vixTermStructure)`
- `bandVvix(vvix.value, config.vvix)`
- `bandVix9dRatio(value, config.vix9dRatio)`
- `bandHyOas(hyOas.value, config.hyOas)`

Purpose: regime-band overrides actually reband the live board.
Output: config-threaded getRegimeBoard + server composition wiring; byte-identical with no overrides.
</objective>
