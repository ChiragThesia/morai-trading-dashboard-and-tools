<objective>
Define the API contract for the runtime rule overrides: a nested, partial, whitelisted Zod
schema that expresses ONLY the curated ~20 knobs (CONTEXT.md "Knob scope"), rejects unknown
keys (`.strict()` at every level), and enforces the two data-integrity invariants at the
boundary â€” scoring weights sum to 100 and hysteresis rung pairs are complete + correctly
ordered.

Purpose: this schema is the single validation seam the PUT route (29-13), the storage repo
(29-08), and the settings use-cases (29-09) all parse through. Getting the whitelist + refines
right here is the primary security mitigation for the phase (V5 Input Validation, T-29 register).
Output: `ruleOverrides`, `getRuleSettingsResponse`, `setRuleOverridesRequest`,
`setRuleOverridesResponse` contracts + their inferred types, exported from the barrel.

Decision (Claude's Discretion, CONTEXT.md): weight-sum is enforced as HARD VALIDATION (reject
when the 9 weights are present and do not sum to 100), not server-side normalization â€” matches
the existing `rules.test.ts` "weights sum to exactly 100" invariant and keeps the stored blob
honest. Weight overrides are all-9-or-none (a partial weight set has no defined sum). Document
this in the schema file's header.
</objective>
