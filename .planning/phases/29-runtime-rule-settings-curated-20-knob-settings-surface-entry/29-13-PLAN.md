<objective>
Expose the settings surface: GET/PUT `/api/settings/rules` (mirroring `journal-rules.routes.ts`)
plus the MCP `get_rule_settings`/`set_rule_overrides` tool pair, sharing the 29-02 contract. Wire
the 29-09 use-cases at the server composition root, injecting the full DEFAULT knob object
computed from the three engines' resolve functions (`resolvePickerRuleConfig()`,
`resolveExitRuleConfig()`, `resolveRegimeRuleConfig()` with no overrides, flattened to the
contract knob shape) — so the displayed defaults never drift from the compile-time constants.

This is the primary trust boundary of the phase: the PUT endpoint mutates live trading rule
thresholds. Mitigations: existing JWT `authReadGroup` middleware (mount inside `apiRouter`),
contract Zod validation on the body (reject unknown keys + weight-sum + hysteresis pair from
29-02), and mapping the use-case `Result` to transport errors (adapters stay thin — parse → call
use-case → map Result → respond).

This plan wires into `apps/server/src/main.ts` AFTER 29-12 (shared file — sequential).

Purpose: the browser + Claude Code can read/update the curated knobs.
Output: settings routes + MCP tools + composition-root wiring.
</objective>
