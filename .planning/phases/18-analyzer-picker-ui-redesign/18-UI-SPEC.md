## Visuals

**Primary focal point: the ranked candidate-cards rail (left column) and the payoff-center
overlay (center column) together** — this is a two-anchor screen, not a single hero. The user's
core loop is: scan ranked cards → click a card to select it (payoff center updates) → optionally
click ⊕ on a second card to compare → read the why-panel/term-structure/plan (right column) for
the selected candidate. Every card must show its rank-relative score at a glance (score number +
4 breakdown bars) without opening anything — the why-panel is progressive disclosure for the
*reasoning*, not required to compare candidates.

**Layout**: 3-column grid, matching `mockups/playground-v4.html` variant B exactly —
`grid-template-columns: 300px 1fr 330px` on desktop (≥1280px viewport, matching the existing
Overview/Analyzer breakpoint convention — no new responsive tier introduced this phase; below
that width the columns stack vertically in DOM order, same graceful-degradation approach already
used elsewhere in `apps/web`, not a new pattern).

- **Left column** (300px): "Suggested calendars" panel, ranked-cards rail (vertical scroll if
  count exceeds visible height — no pagination, matches mockup's flat list).
- **Center column** (flexible): "Risk profile" panel (payoff chart + legend + scenario strip) on
  top, "Scoring methodology" collapsible panel below it.
- **Right column** (330px): three stacked panels — "Why this calendar", "Term structure + your
  legs", "Entry / exit plan".

## Typography

> **INHERITED / FROZEN.** Exactly the same 4-size scale as Phases 9/17/17.1 — this phase
> introduces zero new sizes. **Exactly 2 weights: 400 (regular) and 700 (bold).**

| Role | Size | Weight | Line Height | Font | New elements using it |
|------|------|--------|-------------|------|--------------------------|
| Micro (label / badge / axis tick / breakdown-bar caption / section label) | 9px | 400 | default | JetBrains Mono; section labels Space Grotesk `tracking-[0.09em] uppercase` (distinction is font-family + tracking + uppercase, NOT weight) | Card sub-line (DTE/debit/θ/vega), breakdown-bar labels (`slope`/`fwd edge`/`GEX fit`/`event adj`), event tags (`FOMC◂f`), term-structure axis labels, panel section labels |
| Table / row data | 11px | 400 | default | JetBrains Mono, `tabular-nums` | Scenario-strip cell values, plan-row values (debit/target/stop/manage-by dates) |
| Body / data base | 12px | 400 | 1.45 | JetBrains Mono (`body` default) | Why-panel prose sentences, methodology list items |
| Emphasis value | 14px | 700 | default | Space Grotesk | Card header name (`7500P Aug 8 / Sep 12`), card score number, why-panel stat values (Fwd IV / Slope / Net θ / θ:vega) |

**Exactly two weights: 400 and 700.** No 600 anywhere in this spec. Section labels get their
visual distinction from the Space Grotesk font-family + `tracking-[0.09em] uppercase` treatment
at weight 400 — never a heavier weight. The card header name folds into the **700 emphasis tier**
(it is a primary identifier alongside the score number, not a body label), rather than the
inherited `PositionsTable` 600 convention — this spec declares 700 for it. 700 is reserved
exclusively for the emphasis tier (card header name, card score, why-panel stat values); every
other element — breakdown-bar labels, card sub-line, plan-row labels/values, why-panel prose,
methodology, event tags, scenario-strip values — is weight 400.

## Phase-Specific Component Contracts

Supplementary detail beyond the 6 standard dimensions — this phase replaces a whole screen and
introduces a new contract-typed data shape, so the executor needs an unambiguous rendering
contract per surface.

### Ranked candidate cards (ANLZ-01, D-01/D-05)

- **Card anatomy** (top to bottom, matches mockup `.cand-card`):
  1. Header row: candidate name (14px/700, e.g. `7500P Aug 8 / Sep 12`) + score number
     right-aligned (14px/700, violet).
  2. Sub-line (9px, `--color-dim`): `DTE {front}/{back} · debit ${n} · θ +{n}/d · vega +{n}` +
     inline event tags.
  3. Four breakdown-bar rows (D-05), **rendered from `PickerCandidate.breakdown` array, filtered
     to the 4 criteria named `slope`/`fwdEdge`/`gexFit`/`eventAdjustment`** — never hard-coded
     indices. Each row: 9px label (left, fixed 64px column) → horizontal bar (flex, 5px height,
     track `--color-raise`, fill per the Color section table above, width = `contribution`
     normalized 0–100%) → 9px value caption (right, fixed ~34px column). The 5th breakdown entry
     (BE-vs-EM) exists in the array but is NEVER rendered as a 5th card bar (D-05) — it surfaces
     only in the why-panel/scenario strip.
- **Selected state**: border `--color-violet`, background tint `rgba(167,139,250,.06)` (exact
  mockup value, already a MORAI-palette rgba, not a new token).
- **Hover state**: border `--color-line2` (unselected cards only).
- **Click target**: the whole card (not just the header) selects it as the payoff-center
  candidate — matches mockup's `cand-card` click delegation.
- **⊕ Compare button**: small `button.mini` in the header row (or sub-line), does NOT select the
  card as primary — loads it into the compare slot only. Clicking the currently-compared card's
  ⊕ again clears the compare slot (toggle, matches mockup `CMP=(CMP&&CMP.id===c.id)?null:c`).

### Payoff center: ⊕-compare overlay, EM band, scenario strip (ANLZ-02, D-02/D-02a)

- **Engine reuse (D-02, hard constraint)**: selected candidate → adapt its 2 legs into exactly
  ONE `AnalyzerPosition` (per D-02, a calendar = one position: `frontDte`/`backDte`/`frontIv`/
  `backIv`/`putCall`/`occSymbol` synthesized to carry the strike, `qty` from the candidate,
  `included: true`) → `repriceScenario([position], params)` → feed `payoffCurve` as `todayCurve`
  and `expirationCurve` as the exp curve into `PayoffChart`. **No second payoff code path is
  written.** Compare candidate uses the identical adapter function.
- **Curve color contract**: see Color section (`todayCurveColor="#5b9cf6"`,
  `expirationCurveColor="#a78bfa"`, new `compareCurve`/`compareCurveColor="#f0b429"` dashed).
- **EM band**: two blue tick marks at `spot − candidate.expectedMove` and
  `spot + candidate.expectedMove`, connected by a thin blue horizontal line at the zero-P&L
  y-position, label `±1σ EM` in blue 9px — matches mockup exactly. Additive overlay layer on
  `PayoffChart` (new optional prop, e.g. `expectedMoveBand: { spot: number; em: number } | null`),
  drawn between the existing zero-line layer and the curve layers so it never occludes a curve.
- **Scenario strip** (below the chart, matches mockup): a grid of T+0/@exp P&L values at 5 key
  levels — put wall, γ flip, spot, call wall, and the candidate's own strike. Reuses
  `buildScenarioStrip`'s level-selection logic conceptually (dedup + cap), computed from the
  single adapted `AnalyzerPosition` via the same `repriceScenario`/`bookPL`-style evaluation —
  no new pricing math, just re-running the existing engine at 5 specific spot values instead of
  the full grid.
- **Compare cardinality**: single compare slot only (Claude's Discretion, confirmed default —
  loading a second ⊕ candidate replaces the first, never stacks).

### Why-panel, term structure, entry/exit plan (ANLZ-03, D-01b)

- **Why-panel** (`Stat`-molecule 2×2 grid, matches mockup exactly): Fwd IV (nullable, D-01a) ·
  Slope (violet if positive, down/red if negative) · Net θ (always `--color-up`, constraint
  `>0` enforced by the engine/fixture, never rendered negative) · θ:vega ratio. Below the grid,
  two prose sentences (12px body): forward-edge narrative (candidate-conditional: front-rich vs
  forward-tailwind wording, branches on `fwdEdge > 0`) and event-premium narrative
  (candidate-conditional: warns when `fEvts.length > 0`, otherwise states "structural, not event
  premium"). A third GEX-fit sentence closes the panel (references the fixture's static GEX
  snapshot: flip/walls/netGamma — NOT re-fetched live, this screen scores against the frozen
  D-03 fixture snapshot, matching the mockup's real 2026-07-01 data).
- **Term structure mini-chart** (inline SVG, ~310×150 per mockup, reused proportions): the full
  ATM-IV term-structure line (from the fixture's 31-point curve) + amber dashed vertical event
  markers (FOMC/CPI/NFP) + two leg dots (front = coral/down, back = teal/up, matching mockup) +
  a blue dashed horizontal forward-IV bracket between the two leg x-positions — **omitted when
  `fwdIV` is null** (guard case, see Copywriting Contract) in favor of a `guard` tag.
  Axis: DTE 0–82 on x, IV 8%–15.5% on y (fixture-fixed range, matches mockup — not
  auto-scaled, since the fixture's term structure is static reference data for this phase).
- **Entry/exit plan card**: 5 locked rows (see Copywriting Contract), values read directly from
  `PickerCandidate.exitPlan` (D-01b: `profitTargetPct: 0.25`, `stopPct: 0.175`,
  `manageShortDte: 21`, `closeByExpiry`) — Phase 18 renders these as fixed defaults verbatim, no
  computation beyond `debit × pct` for the target/stop dollar amounts and a date format pass for
  `manageShortDte`/`closeByExpiry`.

### Null-FwdIV / guard-case render branch (D-01a/D-03a/D-06)

Exactly one fixture candidate (D-03a) exercises this branch. Every surface above has a defined
guard behavior — summarized here as the single source of truth for the executor:

| Surface | Normal render | Guard render (`fwdIV === null`) |
|---------|---------------|----------------------------------|
| Card breakdown bar (`fwd edge`) | Bar width from `contribution`, caption `{n}v` | Bar width 0, track color only, caption `n/a` |
| Why-panel Fwd IV stat | `{n}%` value, `vs front {n}%` sub-caption | `—` value, `vs front {n}%` sub-caption (unchanged — front IV is always known) |
| Why-panel forward-edge sentence | Front-rich or forward-tailwind narrative | The dedicated guard sentence (see Copywriting Contract) |
| Term structure | Forward-IV bracket drawn between leg dots | Bracket omitted; `guard` amber tag rendered near the leg dots instead |
| Score | Unaffected — `score` (0–100) is always engine-computed and present; only the `fwdEdge`-named breakdown entry is null-tagged, never the overall score |

No component throws, renders `NaN`, or silently shows a blank number for the guard case — every
surface has an explicit textual fallback (`n/a`, `—`, or the guard tag), per the tagged-`Result`
discipline this contract inherits from the engine's own D-01a requirement.
