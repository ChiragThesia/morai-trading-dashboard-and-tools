<objective>
Extend the existing `scenario-engine.ts` (do NOT fork it) so the payoff engine is calibration-aware
and produces the bounded scenario-strip key-level set. Two pure-function additions, both fully
testable:

1. **Leg-level non-convergence exclusion (Pitfall 1 refines D-02).** The system's instrument is
   always a two-leg calendar. The UI-SPEC's blanket "exclude from T+0, always draw @exp" rule is
   WRONG for the back leg. Correct rule, per RESEARCH Pitfall 1:
   - FRONT leg non-convergent → exclude the position from the **T+0** aggregate only; **@exp still
     renders** (at `frontDte` the front leg is intrinsic — `bsmPrice`'s `T<=0` branch needs no IV).
   - BACK leg non-convergent → exclude the position from **BOTH T+0 AND @exp** (at `frontDte` the
     back leg still has real remaining time value: `backT = max((backDte - frontDte)/365, 1e-6)`).
   Track convergence **per leg** via new `AnalyzerPosition` fields, and have the net-book aggregate
   report how many positions it dropped so the Overview can render "T+0 excludes {n} position(s)".

2. **Scenario-strip key-level set (D-06 / D-07).** A bounded set — put wall, γ flip, spot, call
   wall, plus each position's short/long strikes — deduped, sorted ascending, capped at 8 columns
   (keep the 4 position strikes closest to spot if more remain). The `@exp` column resolves to the
   book's FRONT (nearest) expiry date across included positions (D-07).

This plan owns only `scenario-engine.ts` + its test file — no overlap with Plans 01/03, so it runs
in parallel in Wave 1. It does NOT import `iv-calibration.ts`; it only defines the status fields
that Plan 04 populates from Plan 01's `resolveLegIv` results.

Purpose: Make the payoff math honest about which legs it can and cannot price, and emit the exact
strike set the scenario strip renders.
Output: extended `scenario-engine.ts` + `scenario-engine.test.ts`, TDD red→green, suite green.
</objective>
