<objective>
OPS-02: restructure the compute-bsm-greeks use-case from read-all → solve-all → write-once into a
batch-commit loop, so a full-cohort BSM recompute survives the pg-boss 900s handler cap without the
timeout-and-retry dance. Each batch is a durable checkpoint; the run voluntarily exits with `ok`
before the cap when a wall-clock budget is exhausted, and the remainder resumes on the next trigger.

Purpose: Today one kill anywhere in the run (mid-RTH worker redeploy, OOM, or the 900s expire) loses
every solved row because there is a single writeBsm at the very end. The 19,116-row backlog and the
14:33/14:10 durations brushing the cap (CONTEXT forensics) are this shape. Batching converts "lose
the whole run on any kill" into "lose at most ~800 rows (~1 min)."

Output: computeBsmGreeks.ts batch loop with two named, tested, tunable constants
(COMMIT_BATCH_SIZE, BSM_TIME_BUDGET_MS); durability + budget-exit contract coverage; two corrected
stale lines in jobs.md. No new ports, tables, cursors, or config knobs.
</objective>
