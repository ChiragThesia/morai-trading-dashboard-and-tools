| Role | Value | Usage |
|------|-------|-------|

| Destructive | `text-down` / `bg-downd` (`#ef5350` / `#3e1f23`) | STOP verdict only — same token `MetricChip`'s `alert` prop already uses for negative-gamma/loss emphasis and `LiveStatusBadge`'s STALLED fault state |

Never used for verdict severity — verdict severity is carried entirely by the
up/amber/down semantic triad below, matching Phase 24's regime-board banding lock.

### Verdict severity palette (domain-specific — additive to the 60/30/10 split)

Three visual weights, reusing only existing tokens (no new hex, no new opacity
combinator beyond the `/NN` pattern already used by `bg-violet/10`, `ring-down/40`,
`bg-violet/[0.06]` elsewhere in this codebase):

| Verdict | Weight | Chip treatment | Value color | Existing precedent |
|---------|--------|-----------------|-------------|----------------------|
| HOLD | neutral | plain `MetricChip` (`bg-raise/40 ring-line`) | `text-txt` | Default `MetricChip`, unstyled |
| TAKE (+rung) | neutral-positive | plain `MetricChip` | `text-up` | `checklist-theta` chip (`candidate.theta >= 0 → text-up`, no alert background) |
| ROLL | neutral-caution | plain `MetricChip` | `text-amber` | GEX flip chip / staleness-warning amber dot |
| STOP (−rung) | **escalated** | `alert` `MetricChip` (`bg-downd ring-down/40`) | `text-down`, bold | `MetricChip alert={true}` (negative-gamma "AMPLIFY"), `LiveStatusBadge` STALLED |
| EXIT — pre-event | **escalated** (distinct from STOP) | filled-amber chip: `bg-amber/15 ring-1 ring-amber/40` | `text-amber`, bold | New combinator built from existing tokens only (`bg-amber` + Tailwind opacity modifier, same construction pattern as `bg-violet/10`/`ring-down/40`) — deliberately a different hue at the same "filled" visual weight as STOP, so the two escalated verdicts never read as the same alarm |
| INDICATIVE (AH/stale marks) | display-only, never actionable | `alert` `MetricChip` | `text-amber` value, e.g. `"AH — indicative"` / `"STALE — indicative"` | Exact reuse of `Analyzer.tsx`'s `session-badge` (`ScoringMethodologyPanel`, `marketSession === "after-hours"`) — same `alert` chip + amber value text |

**Escalation-tier rule (CONTEXT.md lock):** STOP and EXIT-pre-event are the only two
verdicts that render on a filled/`alert`-weight chip. HOLD/TAKE/ROLL always render on the
plain (non-alert) chip background — this is the single visual signal for "read this
verdict now" vs. "informational." The two escalated verdicts stay distinguishable from
each other by hue alone (red-fill vs. amber-fill), never by intensity, so a color-blind
scan of "is this urgent" (fill vs. no fill) still works.

**Changed-this-cycle marker (EXIT-09):** no new color. A `changed: true` verdict adds a
small uppercase `"CHANGED"` text label in the SAME color as the verdict's own value color
(e.g., a changed STOP shows `"CHANGED"` in `text-down`, a changed TAKE shows it in
`text-up`) immediately left of the verdict chip. This carries the alert without inventing
a 4th accent — matches the "reuse existing badge/banner conventions" lock and avoids
touching the violet-reserved rule.

## Component & Layout Contract

*(Additive detail beyond the template — the specific decisions the planner/executor need that don't fit the generic table shape above.)*

### Placement (locked: "Analyzer screen: held-positions panel")

New full-width `<Panel>` section on `Analyzer.tsx`, placed **below** the existing 3-column
grid (`Suggested calendars` / `Risk profile` / `Why this calendar`) — a new sibling
section, not inserted into the candidate-picker's own columns. This keeps PICK-02's "no
layout change" guarantee for the existing rail/payoff/why-panel columns (held positions
are a different data source — open calendars, not scored candidates — so they get their
own row, matching Phase 24's precedent of adding a new full-width section rather than
squeezing new content into an already-tuned grid).

```
[ ScoringMethodologyPanel strip — unchanged ]
[ 3-col grid: rail | risk profile | why-panel — UNCHANGED ]
[ NEW: "Held positions" panel — one row per open calendar ]
[ NEW: "Exit rules" panel — the registry, collapsible like ScoringMethodologyPanel's checklist ]
```

### Held-position row anatomy (per open calendar)

One row per open calendar inside the `"Held positions"` `Panel`, styled like a compact
`CandidateCard` (border-line rounded row, `px-2.5 py-2`, NOT selectable/clickable — this
is a display-only advisory list, no click-to-select since there is nothing to select into):

1. **Header line:** calendar name/strike/expiries (`text-sm font-bold text-txt`, mirrors
   `CandidateCard`'s name row) + the `"CHANGED"` marker (if `changed: true`) + the verdict
   chip (`MetricChip`, severity-colored per the Color contract) on the same row,
   right-aligned — mirrors `CandidateCard`'s `name ↔ score` header pattern.
2. **Rule/metric line:** `text-[9px] text-dim font-mono`, the exact copy from the
   Copywriting Contract's "Rule/metric line" row — mirrors `EntryExitPlan`'s footnote
   styling.
3. **Session/staleness dot:** reuses `CandidateCard`'s `size-1.5 rounded-full` dot +
   `formatAsOf`-style label pattern; when `indicative: true`, the row's verdict chip is
   FORCED to the INDICATIVE treatment (`alert` chip, amber value, "AH — indicative" /
   "STALE — indicative") regardless of what the underlying verdict would otherwise be —
   an indicative STOP/TAKE never renders in its escalated STOP/TAKE colors (locked:
   "never styled as actionable").
4. **ROLL detail row** (only when `verdict === "ROLL"`): one extra `PlanRow`-style line
   with the suggested front expiry + estimated debit.

`data-testid` convention (extends `candidate-card-{id}` / `checklist-{key}` precedent):
`held-position-{calendarId}`, `held-position-verdict-{calendarId}`,
`held-position-rule-{calendarId}`, `held-position-changed-{calendarId}` (present only
when changed), `held-position-indicative-{calendarId}` (present only when indicative),
`held-position-roll-{calendarId}` (present only when verdict is ROLL).

### Exit rules panel (EXIT-07 — entry-methodology symmetry)

A second `Panel` titled `"Exit rules"`, rendering the snapshot's `ruleSet` payload
(mirrors `ScoringMethodologyPanel`'s reliance on the engine's own registry — never a
client-side copy). Unlike the picker's weight-relative score checklist, exit rules are a
precedence ladder of gates/triggers with no score weight, so the rendering is a compact
list, not scored chips:

- One row per `ruleSet` entry: `SectionLabel`-style 10px label (rule id/kind, e.g.
  `"STOP · risk"`) + `text-[10px] text-dim font-mono` rationale sentence (the registry's
  own `rationale` field, verbatim — never hardcoded UI prose per BOARD-02's "payload-carried
  copy" precedent applied here).
- Rows render in the engine's own precedence order (the payload's array order — the
  registry is the single source of truth for ladder position, the UI never re-sorts).
- No collapse/expand interaction needed at this row count (7 rules: STOP/EVT/GAMMA/TERM/
  TAKE/ROLL/HOLD) — renders flat, unlike the picker's `ScoringMethodologyPanel` which
  hides behind a checklist-chip summary. If the row count grows, follow the picker's
  collapsible pattern; not needed for this phase's fixed 7-row registry.

### States

- **Loading** (`isPending && data === undefined`): single centered message inside the
  `"Held positions"` `Panel`, see Copywriting Contract → Loading state.
- **Error**: centered message + `Retry` button, see Copywriting Contract → Error state.
- **Cold-start** (open calendars exist, snapshot is `null`): see Copywriting Contract →
  Cold-start state.
- **Empty** (zero open calendars): see Copywriting Contract → Empty state. This is
  distinct from cold-start — empty means "nothing to advise on," cold-start means
  "advice is coming."
- **Loaded:** one row per open calendar, in the payload's own array order (no
  client-side re-sort — the use-case's own read order is the display order).

### Interaction

- Held-position rows are NOT clickable/selectable (no `onSelect` — this list has no
  downstream selection target, unlike `CandidateCard`). Hover state is a subtle border
  lighten only (`hover:border-line2`, matches `CandidateCard`'s non-selected hover), no
  cursor-pointer.
- No destructive actions, no order-entry affordances anywhere in this panel (EXIT-10 —
  advise + alert only).
- The `"CHANGED"` marker and escalated (STOP/EXIT-pre-event) chip fills are the ENTIRE
  alert surface for EXIT-09 — no toast, no fixed-bottom banner, no browser notification.
  Confirmed no toast system exists in `apps/web/src` today (RESEARCH.md); a per-row
  marker at this weight matches the existing `LiveStatusBadge`/`AuthExpiredBanner`
  scoping precedent (fixed banners reserved for account-wide states, not per-position
  concerns).
- Exit-rules panel rows have no interaction (read-only reference, like a doc table
  rendered in-app) — no tooltip needed beyond the rationale text already inline.
