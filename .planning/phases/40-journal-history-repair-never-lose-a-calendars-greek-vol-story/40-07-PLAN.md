<objective>
Ship HIST-04's operator repair path in the shape the autonomous run needs: a one-time full-history rebuild for one or all calendars, runnable in prod WITHOUT direct psql. Per the orchestration constraint, the repair is an on-demand pg-boss job (register-open-calendars pattern) the orchestrator can enqueue via the existing `trigger_job` MCP tool after deploy, plus a thin operator CLI (fix-pnl-reingest pattern) for railway-run use. Both call one core orchestrator use-case that reuses the plan-05 rebuild engine and prints per-calendar before/after coverage counts (D-10).

Two safety rules from RESEARCH's Security Domain + D-08:
- The default run is heal-only (fill-only rebuild — never destructive). The post-close/pre-open TRIM (delete outside the life window) is opt-in, CLI-only, and prints the deleted count. `trigger_job` can heal one or all, but can never trim.
- The CLI's "all calendars" mode is explicit (no silent default-to-all); every step is idempotent so a partial failure is safely re-runnable.

Plus HIST-04's on-register backfill: registering a calendar triggers a backfill from `openedAt`, non-fatal to the registration (re-runnable), so late registration never loses the entry-day story.

The `repair-journal-history` Job Catalog row already landed in plan 01 (docs-before-code). The actual prod repair RUN is a post-deploy operator/orchestrator step in the plan 08 gate — NOT executed here.

Purpose: The mechanism to repair all 17 existing calendars, triggerable without shell access.
Output: `repairJournalHistory` use-case + job + CLI + on-register backfill + TRIGGERABLE_JOBS entry.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator/orchestrator → repair trigger | An "all calendars" or a destructive trim run must be intentional, bounded, and re-runnable |
| trigger_job payload → repair scope | An arbitrary job name or an unbounded destructive flag must not reach the repair |
| newly-registered calendar → backfill | A backfill failure must not corrupt or block the registration |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-40-04 | Repudiation / accidental data loss | repair CLI "all" + trim | high | mitigate | CLI requires an explicit --all (no silent default-to-all) and gates deletion behind --trim with a printed count; every step idempotent so a partial failure is safely re-runnable (RESEARCH Security Domain + D-08) |
| T-40-15 | Tampering (destructive via remote trigger) | trigger_job → repair | high | mitigate | trigger_job path never exposes the trim flag — heal-only via remote; destructive trim is CLI-only, matching D-08's "post-close rows may be deleted only by the explicit CLI repair" |
| T-40-06 | Elevation / Input Validation | repair job payload + CLI argv | medium | mitigate | z.enum(TRIGGERABLE_JOBS) rejects arbitrary names (T-05-22); handler Zod-parses { calendarId?, trimOutsideWindow? }; CLI Zod-parses argv per typescript.md |
| T-40-16 | Availability (registration integrity) | on-register backfill | medium | mitigate | Backfill is non-fatal to registration; a failure leaves the calendar registered and the history re-runnable (never a half-registered state) |
</threat_model>
