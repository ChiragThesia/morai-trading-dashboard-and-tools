<objective>
Fix HIST-05's series inflation (~19 vs ~13 rows/day) at the write-side key, not the schedule. RESEARCH traced the duplicate scheduled rows to a collision: the 30-min `fetch-schwab-chain` chain-trigger and the independent hourly `compute-bsm-greeks` fallback cron both terminate at `snapshot-calendars` near the top of each hour, 10-15 min apart, stamping two rows with different exact wall-clock `time` values against a `(time, calendar_id)` composite PK — so `onConflictDoNothing` never collapses them.

The fix rounds a `scheduled`-trigger row's `time` down to its nominal 30-min RTH slot boundary before persisting (using `roundDownToRthSlot` from plan 01), so the second same-slot write's key collides with the first and the EXISTING `onConflictDoNothing` absorbs it — zero new dedup logic, no cron change (the hourly fallback keeps its OPS-02 resilience role). `event-move` rows keep their real timestamp (D-07), and the OPS-01 freshness gate still runs against the real clock. This is a behavior change to the live writer, so it lands behind a failing test first.

Purpose: One scheduled row per 30-min slot per calendar, reusing the constraint already in place.
Output: Slot-rounded `scheduled` snapshot timestamps.
</objective>
