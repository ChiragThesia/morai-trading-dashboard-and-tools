<objective>
Build the two new driven ports HIST-02's rebuild use-case needs, plus the D-08 trim port the operator repair needs — each with its Postgres implementation, in-memory twin, and contract tests (architecture-boundaries rule 8). RESEARCH established that the existing ports cannot satisfy this phase: `ForResolvingLegSnapshot` resolves only the LATEST observation (not as-of a historical slot), and `ForPersistingSnapshot` is INSERT-ONLY (`onConflictDoNothing`), so D-03's "replace a gap row with a healed one" is structurally impossible with it (Pitfall 2).

Three ports:
- `ForResolvingLegObservationForSlot` — as-of-slot read, mirroring `readSmile`'s "latest at-or-before an anchor" two-step, root-candidate-aware (reuses plan 01/02's fix). The usability bound reuses the live freshness window (D-07) so a slot with no usable observation stays an honest gap (D-04).
- `ForHealingSnapshot` — fill-only conditional write. SELECT-then-decide inside a transaction (mirroring `recomputeSnapshotPnl`), reusing the LOCKED `isGapRow` from `attribution.ts` — never a second gap definition.
- `ForDeletingSnapshotsOutsideWindow` — deletes only rows outside `[openedAt, closedAt]` for a calendar, returning the count (D-08; used only by the operator repair, plan 07).

Architecture rule 9 (new use-case ⇒ HTTP route + MCP tool same PR) is WAIVED for this phase per D-01 — these ports back a CLI + job, not a new read API.

No schema change: `calendar_snapshots`'s composite PK `(calendar_id, time)` already supports the heal UPDATE and the windowed DELETE (RESEARCH confirmed). This plan adds NO migration.

Purpose: The storage primitives the rebuild use-case (plan 05) and repair orchestrator (plan 07) compose.
Output: Three ports + Postgres + memory twins + contract tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| rebuild/self-heal → calendar_snapshots write | A healed row must never overwrite a genuinely live (non-gap) row |
| operator repair → windowed delete | A delete must never remove an in-window (life-of-trade) row |
| query inputs (strike/expiry/root) → SQL | Hand-built strings could reach a raw query |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-40-02 | Tampering (data integrity) | ForHealingSnapshot | high | mitigate | Fill-only via the SAME isGapRow predicate, inside one transaction so the live writer is never clobbered by a read-then-write race (D-03); contract test proves no-op-when-live |
| T-40-12 | Tampering (accidental data loss) | ForDeletingSnapshotsOutsideWindow | high | mitigate | Delete predicate is strictly time < openedAt OR time > closedAt; contract test proves in-window rows are preserved and the returned count is exact (D-08) |
| T-40-05 | Tampering | as-of-slot read query construction | medium | mitigate | Roots from resolveRootCandidates's closed union; parameterized Drizzle (inArray/eq/lte), never interpolated strings |
| T-40-03 | Tampering (honest-gap) | as-of-slot read usability window | medium | mitigate | An observation older than the freshness window returns null (honest gap, D-04) rather than a stale value dressed as fresh |
</threat_model>
