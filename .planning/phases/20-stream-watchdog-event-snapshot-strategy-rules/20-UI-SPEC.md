# Phase 20 — UI Design Contract

> Visual and interaction contract for WATCH-01 + RULE-01. **NOT a redesign.** SNAP-01 is
> server-side/headless (D-04) and has **zero UI surface** — do not invent UI for it. WATCH-01
> modifies one existing presentational component (`LiveStatusBadge.tsx`) in place, 4-state →
> 3-state. RULE-01 adds one new control to an existing panel (`Journal.tsx`'s "thesis · review"
> Notes card) plus a compact read-view tag on the existing trade-list rows. Both reuse the
> Phase-21 `Button` system primitive and the locked MORAI token palette — zero new hex values,
> zero new npm dependencies, zero new shadcn components. Generated by gsd-ui-researcher, verified
> by gsd-ui-checker.

## Scope Statement (read first)

Three sub-items, two touch UI:

1. **WATCH-01** — `LiveStatusBadge.tsx` goes from 4 states (`live`/`stale`/`reconnecting`/`poll`)
   to 3 (`LIVE`/`QUIET`/`STALLED`), server-driven (D-01/D-03). This spec locks the exact tone
   distinction the discuss-phase deferred here: **STALLED must read as a genuine alarm — louder
   than the old amber "stale" look** (D-20). Also specifies the cold-start "connecting" treatment
   (D-11) and the manual force-reconnect action (D-17).
2. **RULE-01** — a new multi-select rule-tag control keyed to event type (ENTER/EXIT/ROLL, D-07)
   lives inside the existing Notes ("thesis · review") panel in `Journal.tsx`, next to the
   existing free-text textarea (D-10, kept as optional supplement — unchanged). Recorded tags
   also render as a compact read-only chip in the left-column trade-list rows (D-22).
3. **SNAP-01** — **no UI**. Detection and the ad-hoc `boss.send` enqueue are entirely server-side
   (D-04). If a plan or executor finds itself adding a snapshot-triggered toast, banner, or badge,
   that is out of contract for this phase.

No changes to: Journal's 3-column grid, panel set, snapshot table, lifecycle chart, or any
Overview/Analyzer surface from Phases 17-19.

## Visuals

**No new focal point, no new screen.** Two small, existing-shell modifications:

- **Stream badge** (wherever `LiveStatusBadge` renders today — Overview header) keeps its exact
  position, size, and Badge/Tooltip shell. Only the per-state color/label/dot table and tooltip
  copy change, plus a new inline action (force-reconnect) that appears only in the STALLED state.
- **Journal Notes panel** gains a compact rule-tag row of toggle chips above the existing
  textarea — same `Panel`/`PanelHeading` shell, same card, no new panel, no layout-width change.
  Each event-type section (ENTER always, EXIT once closed, ROLL per roll event) is a labeled row
  of small toggle chips using the Phase-21 `Button` `variant="toggle"` primitive verbatim.
- **Trade-list rows** (left column) gain one more small neutral pill — same visual grammar as the
  existing `history`/`entry/exit`/`OPEN` pills already in that row — shown only when the trade has
  ≥1 recorded rule tag.

## Color

> **INHERITED / FROZEN 60/30/10 split.** Zero new hex values. Every new state/tag reuses an
> existing MORAI token by role — this is the entire point of D-20's deferred decision.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0a0e14` (`--color-bg`) | Page background — unchanged |
| Secondary (30%) | `#0f1521`/`#161d2b` (`--color-panel`/`--color-raise`) | Panel surfaces, tag/chip backgrounds (`bg-raise`) — unchanged |
| Accent (10%) | `#a78bfa` (`--color-violet`) | **Unchanged reservation** (selected-trade border/`bg-violetd`, focus ring, primary Button, toggle-active fill). **New usage, same reservation, not a new item**: RULE-01's rule-tag toggle chips reuse `Button variant="toggle" tone="violet"` — this is the Phase-21 `Button` primitive's own default active-fill behavior, not a new color decision. |
| Destructive/alarm | `#ef5350` (`--color-down`) on `#3e1f23` (`--color-downd`) | **Unchanged token, reused for a new state**: WATCH-01's STALLED badge (D-20 resolution — see below). Already the app's "genuine problem" semantic (negative P&L, `MetricChip alert` background, destructive Button hover) — reusing it here is what makes STALLED read as a real alarm rather than another shade of the existing amber "stale" look. |

**D-20 resolution — STALLED tone (the deferred decision this UI-SPEC exists to answer):**

The old 4-state badge used **amber** (`#f0b429`) for `stale`. Phase 20's STALLED reuses the
**down/red** alarm token instead (`text-down` on `bg-downd`, `ring-down/40`) — the exact visual
grammar `MetricChip`'s `alert` prop already uses for "negative gamma / loss emphasis" elsewhere in
the app. This is a deliberate escalation: amber stays reserved for "worth flagging, not broken"
(e.g. Phase 19's GEX/events-unavailable tags); red is reserved for "something is actually wrong."
STALLED folding both "ticks frozen" and "transport dead" into one state (D-01) means the badge
must never look merely stale again — it must look like a fault.

| Badge state | Text | Background | Ring | Dot | Rationale |
|---|---|---|---|---|---|
| LIVE | `text-up` (`#26a69a`) | transparent | none | pulsing `.live-dot` (reused verbatim) | Unchanged from today's `live` state |
| QUIET | `text-dim` (`#566273`) | transparent | none | none | Benign — matches today's `poll` state exactly; "the market just isn't open" |
| CONNECTING (transient, D-11) | `text-dim` (`#566273`) | transparent | none | none | **Same exact classes as QUIET** — D-11 explicitly requires reusing QUIET's styling for cold-start/mid-reconnect-during-RTH so there is never a false red on page load. Only the label/tooltip text differs (see Copywriting) — this is a text-only distinction, not a 4th visual state, so the badge component's public state prop stays 3-valued (D-01). |
| STALLED | `text-down` (`#ef5350`) | `bg-downd` (`#3e1f23`) | `ring-1 ring-down/40` | none (a static ring, not a pulse — a pulsing red dot risks feeling decorative like the LIVE dot; a filled alarm background communicates "stopped," not "still happening") | D-20 resolution — see above |

Accent reserved for: selected-trade border/background, focus rings, primary Button, toggle-active
fill (unchanged) **plus** RULE-01's rule-tag toggle-chip active state (same primitive, no new
reservation). No second semantic/destructive color introduced — `down` already covers this role.

## Phase-Specific Component Contracts

### WATCH-01 — `LiveStatusBadge.tsx` (3-state, D-01/D-02/D-03/D-11/D-17/D-20)

- **Public prop** stays a single `status` field, now `"live" | "quiet" | "stalled"` (3-valued,
  replacing today's 4-valued `LiveStreamStatus`). `useLiveStream.ts` derives this from the
  server-emitted heartbeat RTH flag (D-03) + the ~20s stall timer (D-02), never a local clock.
- **CONNECTING is not a 4th prop value** — it is a client-side rendering condition: `status ===
  "quiet"` AND the RTH flag on the current heartbeat is `true` (market IS open, just no tick yet).
  The component computes label/tooltip from `(status, isRthAccordingToLastHeartbeat,
  hasReceivedFirstTick)`, not from a new enum member, so the state machine itself stays exactly
  3-valued per D-01.
- **Dot**: only LIVE shows the pulsing `.live-dot` (reused verbatim from `index.css`, unchanged
  keyframe). QUIET/CONNECTING/STALLED show no dot.
- **Force-reconnect button (D-17)**: rendered inline, immediately after the badge, ONLY when
  `status === "stalled"`. On click: mints a fresh single-use ticket and reconnects immediately
  (reusing `useLiveStream`'s existing `connect()` path) AND cancels the pending exponential-backoff
  reconnect timer first (D-17's explicit double-connect guard) before opening the new connection.
  Button is disabled (not hidden) while a manual reconnect is already in flight, label changes to
  `Reconnecting…` during that window (mirrors the existing `Rebuilding…` pending-label convention
  from `RebuildButton.tsx`).
- **Tooltip** keeps the existing `TooltipProvider`/`Tooltip` shell; only per-state copy changes
  (see Copywriting Contract table above). STALLED tooltip additionally surfaces the force-reconnect
  affordance is present (the button sits next to the trigger, not inside the tooltip content —
  tooltips are not interactive in this codebase's existing pattern).
- **Guard**: never renders a blank or `undefined` label — if the heartbeat payload fails Zod
  parsing, the badge holds its last-known-good state rather than flipping to an unstyled default
  (matches the "drop malformed, never cast" rule already enforced in `useLiveStream.ts`).

### RULE-01 — Strategy rule control (D-07/D-08/D-09/D-10/D-13/D-14/D-21/D-22)

- **Placement (write/edit surface)**: inside the existing Notes `Panel` in `Journal.tsx`
  (`PanelHeading title="Notes" action={<HeadingPill>thesis · review</HeadingPill>}`), as a new
  block ABOVE the existing free-text `<textarea>`. The `textarea` itself is untouched (D-10:
  optional supplement, unchanged placeholder/styling).
- **Anatomy per section** (one row per applicable event type):
  - `SectionLabel` (`ENTER`/`EXIT`/`ROLL`), reused verbatim (10px/600/uppercase/tracked, `text-dim`
    tone) — matches every other section label in the app (`Panel`/`Stat`/`MetricChip` precedent).
  - A flex-wrapped row of toggle chips, one per enum value for that event type, using
    `Button variant="toggle" tone="violet" size="xs"` verbatim — `active` = tag currently recorded
    for this event. Clicking toggles that single tag; multiple can be active at once (D-14:
    list-shaped, multi-select).
  - The `Other` value is the last chip in every row. When active, an inline single-line text input
    appears directly below that section's chip row (not a modal, not a separate panel) — required
    before the tag set can be considered saved (D-21). Deactivating `Other` hides the input and
    discards its (now-irrelevant) text.
- **ENTER** section always renders (every trade has an OPEN event). **EXIT** section renders its
  chip row once the trade has a CLOSE event; before that, shows the muted placeholder line
  (`Available at close.`) instead of chips — never shows disabled/grayed-out chips. **ROLL**
  section renders once PER actual ROLL event on the trade (0, 1, or many) — each with its own
  independent chip row and its own timestamp label (`{fmtDate(rollEvent.occurredAt)}`) so multiple
  rolls on one trade are distinguishable.
- **Editable anytime** (D-10): every section's chips remain clickable regardless of trade state
  (open/closed) — there is no locking/read-only mode in this control.
- **Read view (D-22)**: in the left-column trade-list row (`Journal.tsx` trade-list map, the block
  currently rendering the `history`/`entry/exit` kind badge), append ONE more small pill — same
  `rounded-[3px] border px-[5px] text-[8px]` grammar as the existing History badge, `border-line2
  text-dim` (neutral, NOT violet — read-view tags stay neutral/informational; violet stays
  reserved for the live editing-control's active state per the Color contract) — shown only when
  the trade has ≥1 recorded rule tag across any of its events. Content: the tag labels
  comma-joined, truncated with `…` if the row would overflow (exact truncation width is an
  executor implementation detail, not a new design decision). This lets a user "scan which rule
  fired across trades at a glance" per D-22 without opening each trade.
- **Adapter surface (D-13)**: the control's save action calls a new HTTP route (list-shaped
  request/response, D-14) — on failure, render the `Rule-tag save error` copy (see Copywriting)
  inline below the affected section, with a `Retry` mini-button that resubmits the same payload.
  No optimistic-then-silently-revert pattern — the chip's visual `active` state only flips after
  the save call resolves successfully (avoids a UI that lies about what's actually recorded, the
  same honesty principle WATCH-01 enforces for stream state).

### SNAP-01 — Event-triggered supplemental snapshot (D-04/D-05/D-06/D-12/D-15)

**No UI surface.** Detection, debounce, and the ad-hoc `boss.send('snapshot-calendars', …)`
enqueue are entirely server-side (D-04). The only artifact visible to a user is the resulting
snapshot ROW itself, which already renders in the existing `SnapshotTable` (Journal right column)
exactly like a scheduled snapshot — no new column, badge, or visual distinction is specified for
this phase. (The nullable `trigger`/`reason` marker from D-12 exists for future filtering/analysis,
not for display — if a later phase wants to surface "event-move" provenance in the UI, that is a
new UI-SPEC decision, not implied by this one.)
