# Phase 35.1 — Validation Strategy

## Test Infrastructure

| Property | Value |
|----------|-------|
| **Framework** | Vitest 4.1.8 + @testing-library/react 16.3.0, jsdom (apps/web) |
| **Config** | `apps/web/vitest.config.ts` — existing, unchanged |
| **Quick run command** | `bunx vitest run apps/web/src/screens/Overview.test.tsx` (or touched file) |
| **Full suite command** | `bun run test` (root workspace) |
| **Structural limit** | jsdom has NO layout engine: bleed/wrap/overflow/paint claims are jsdom-blind. Automatable = DOM structure, classes, attributes, order (compareDocumentPosition), component behavior, prop defaults. Layout claims go through the chrome-devtools checklist (C1–C12). |
| **Branch mechanism** | `useIsDesktop` is FALSE under jsdom default → mobile tree is jsdom's default render. Desktop assertions require the matchMedia stub (pattern exists from Phase 35 MarketRail tests). Existing Overview.test.tsx desktop tests MIGRATE to the stub in the same plan as the Overview switch (J2 — the byte-identity guard). |

## Sampling Rate

- **Per task commit:** `bunx vitest run <changed test file>`
- **Per wave:** `bun run test` (full workspace)
- **Phase gate:** full suite + typecheck + lint green, then chrome-devtools 390×844 +
  320px spot + 1440px tripwire checklists (C1–C11), then user phone check (C12 — the
  only bar that matters).
