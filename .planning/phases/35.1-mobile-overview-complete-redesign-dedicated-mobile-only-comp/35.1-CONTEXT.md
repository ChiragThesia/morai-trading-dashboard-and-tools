# Phase 35.1 Context — Mobile Overview Complete Redesign

## Why (user-locked)

Phase 35's responsive re-composition (same desktop DOM, `lg:` class branches) produced
"desktop panels reflowed", not a mobile design. User verdict on phone: still garbage.
User explicitly authorized what Phase 35 forbade: **dedicated mobile-only components**.
Scope discipline: **one flow, done completely** — Overview (the flow both complaint
screenshot sets came from, the most-checked screen).

## Ground truth — what still fails at 390px (agent screenshots 2026-07-11, live bundle)

1. **Chip strips are still a wall of pills.** 4-chip row + 6-chip snap rail = two rows
   of desktop pills. No hierarchy — BOOK +$142 (the answer to "how am I doing") is the
   4th chip at the same visual weight as FED FUNDS.
2. **Chart panel is desktop chrome shrunk.** Panel title block + two freshness chips
   (one wrapping) + date-picker rail (arrows/date/Today/Fan/@exp/W… clipped) + 5-item
   legend + 2 rows of BE pills — ~5 rows of chrome before any chart. Chart itself sits
   inside panel padding, not full-bleed.
3. **MARKET section = desktop panels at phone width.** Dealer Γ profile renders as
   near-empty crosshair lines; GEX-by-strike is axis soup with one dashed spot line.
   These are desktop visualizations that don't survive 390px — they need purpose-built
   compact forms (or key-numbers-only), not shrinking.
4. **Position cards are table rows restyled** — functional (best part of 35) but flat:
   label/verdict hierarchy weak, NET VAL/UNREAL as a 2-col label grid.

## Deliverable

A **mobile-first Overview** built as a dedicated component tree (e.g.
`apps/web/src/screens/mobile/` or `components/mobile/`), rendered below the `lg:`
/1024px cutover via the existing `useIsDesktop` hook (or CSS-pairing where DOM
duplication is cheap). Desktop Overview code path stays byte-identical in render
output at ≥1024px.

Design intent (UI-SPEC owns the final contract):
- **Hero-first**: BOOK P&L as the large focal number with SPX/VIX context inline —
  answer "how am I doing" in one glance, Robinhood-hierarchy, not a chip parade.
- **Chart**: full-bleed width, chrome collapsed to one compact control row; date/series
  toggles behind progressive disclosure; BE numbers stay (user-loved feature).
- **Positions**: cards refined (hierarchy: strike/verdict → P&L → detail on expand).
- **Market**: purpose-built mobile form — key levels list works; Dealer Γ + GEX-by-strike
  either get compact mobile-fit visuals or reduce to their headline numbers behind a tap.
- **Ticker chips**: demoted to a single compact strip or into hero/market — no 2-row wall.

## Constraints (locked)

- Mobile-only components ARE allowed and expected (user's words). Duplication of view
  code between desktop and mobile trees is acceptable; duplication of data/logic is not
  — data hooks, scenario engine calls, formatting utils stay shared.
- Data layer untouched. Chart INTERNALS (PayoffChart/scenario engine, Phase 33/34 stack)
  untouched — container sizing/chrome/composition may change; a mobile wrapper may pass
  different props (e.g. hide legend, different margins) if props already exist or are
  pure additive presentational props.
- MORAI design system stays (tokens, mono type, tracking scale, Panel/Button/ChipRail
  primitives available for reuse where they fit).
- No new dependencies.
- Desktop ≥1024px: zero visual regression (tripwire checklist from 35 reusable).
- `useIsDesktop` exists (matchMedia 1024px, jsdom-safe false) — the sanctioned branch
  mechanism; catch #24 law applies (structural open/closed states need real attributes,
  not CSS reveals).
- Analyzer/Journal mobile: OUT of scope (keep Phase 35 state) — one flow, complete.
- UI-SPEC gate before implementation (UI phase discipline). 35-RESEARCH.md mobile-UX
  patterns remain valid input; re-research of generic mobile UX not needed — the spec
  should spend its effort on THIS screen's concrete design.
