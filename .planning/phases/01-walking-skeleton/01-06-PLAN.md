|----------|-------------|

| GitHub repo → Railway | Push to main triggers auto-deploy (D-01) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-15 | Information Disclosure | Secrets committed to the repo (DATABASE_URL, MCP_BEARER_TOKEN) | mitigate | `.env.example` carries placeholders only; real secrets live only in Railway/Supabase config; verify task scans for committed connection strings/tokens |
| T-01-16 | Tampering | DB connection string exposure via the 6543 pooler / wrong mode | mitigate | DEPLOY-01 wires the Supabase DIRECT/session connection (required for migrations + future pg-boss); pooler explicitly not used (deployment.md, D18) |
| T-01-17 | Spoofing / Elevation of Privilege | Unauthenticated MCP access in production | mitigate | Bearer enforced on `/mcp` (verified live: bad bearer → 401) — DEPLOY-03 |
| T-01-18 | Denial of Service | Missing env var → Railway infinite restart loop | mitigate | Config logs a clear message naming the var before `process.exit(1)` (RESEARCH Pattern 7); deploy verification catches a stuck service |
| T-01-SC | Tampering | CI installs (setup-bun action, frozen lockfile) | accept | `bun install --frozen-lockfile` pins the audited lockfile from plans 01-05; no new app packages introduced here |

> Note: `broker_tokens` / Schwab secrets + `TOKEN_ENCRYPTION_KEY` are OUT OF SCOPE this phase (Phase 4). The only live secrets in Phase 1 are `DATABASE_URL` and `MCP_BEARER_TOKEN`.
