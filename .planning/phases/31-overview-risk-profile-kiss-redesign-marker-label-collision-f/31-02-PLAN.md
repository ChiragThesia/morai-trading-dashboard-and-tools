<objective>
Fix defect DEFECT-2: the left-rail MARKET REGIME rows print raw values (VIX/VIX3M, VVIX,
VIX9D/VIX, HY OAS) and force the reader to remember each warn/crisis band. Replace each of
the 4 banded rows with a compact linear bullet gauge — a value marker on a warn/crisis-banded
track — so proximity to the edge reads at a glance (user request 2026-07-10; gauge style
locked as linear bullet, NOT circular dials).

The band edges must be the EFFECTIVE thresholds (Phase-29 overrides-aware), which are already
computed server-side in getRegimeBoard.ts via resolveRegimeRuleConfig() but discarded before
the response is built. This plan threads them out as two additive REQUIRED contract fields
(bandWarn/bandCrisis) — single source of truth, no client-side threshold copies. Required (not
optional) so a stale post-deploy response fails loud via Zod rather than rendering undefined
gauge math (RESEARCH Pitfall 2 / Security V5).

Because the fields are required, adding them breaks every existing regimeIndicator fixture in
the repo (contracts test, server route test, web component + rail tests) in the same wave — so
this one plan owns the whole vertical (contract → core → web) to keep the suite green; splitting
it would force two plans to edit the same fixture files for no gain (ponytail).

The gauge visual, ARIA, scale table, and "what does NOT change" list follow the UI-SPEC design
contract (Component Specification §2 and §3). RESEARCH is authoritative for the file:line push
sites and the reuse pattern (CandidateCard bar-track); where UI-SPEC and RESEARCH differ on the
visual (UI-SPEC: banded track + role=meter + specific tokens), UI-SPEC wins.

Purpose: macro rail reads at a glance; thresholds stay single-sourced from Phase-29 config.
Output: additive contract fields, core population, and the RegimeGauge component.
</objective>
