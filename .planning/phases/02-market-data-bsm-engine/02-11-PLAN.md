<objective>
Close UAT Gap B: live `GET /api/status` returns 500 with a ZodError
`invalid_format "Invalid ISO datetime"` at `lastJobRuns.*.lastErrorAt` /
`lastSuccessAt`. Root cause: postgres.js returns `pgboss.job.completed_on` (timestamptz)
as a Postgres TEXT string like `2026-06-12 13:31:38.031+00`. The contracts
`jobRunRecord` schema (`packages/contracts/src/status.ts`) requires a Z-anchored
ISO-8601 datetime (`z.string().datetime()`), so the text form fails parsing at the
route boundary.

In `job-runs.ts`, `extractCompletedOn` already calls `.toISOString()` when the value is
a `Date`, but passes a `string` straight through — and production returns the string form,
bypassing normalization. Fix: normalize the string case to ISO-8601 Z via
`new Date(value).toISOString()` (guarding `Invalid Date`) before the value crosses the port.
Contracts schema stays strict — no loosening.

A second, equally important fix: the existing job-runs contract test never created a
`pgboss` schema, so `readJobRuns` always hit the Pitfall-6 fallback and returned `ok({})`.
The real-timestamp path was NEVER exercised in tests — that is exactly why this shipped
broken. Add a contract test that seeds a minimal `pgboss.job` row and asserts the returned
record parses against the contracts schema.

Purpose: make `/api/status` return 200 with populated `lastJobRuns`, and close the test
blind spot that let the timestamp defect reach production.
Output: normalized repo output + a real-pgboss-row contract test.
</objective>
