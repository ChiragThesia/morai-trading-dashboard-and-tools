<objective>
The CBOE chain vertical slice: a delayed SPX/SPXW option chain flows from the CBOE CDN
through Zod parsing, OSI→OCC conversion, near-term/near-money filtering, and append-only
persistence into `leg_observations` tagged `source='cboe'` with `bsm_*` left NULL, plus
first-seen `contracts` metadata. End-to-end (port → adapter → use-case → repo) so after
this plan a real chain can be fetched and stored. Implements MKT-01 + MKT-03.

Purpose: This is the data intake half of Phase 2 — without stored cboe rows there is
nothing for the BSM compute job (Plan 06) to fill. It is a vertical slice per MVP mode:
adapter + in-memory twin + repo + use-case shipped together, each contract-tested.
Output: ForFetchingChain port, CBOE + in-memory chain adapters (contract-tested),
leg-observations persistence repo (testcontainers), fetchChain use-case.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Tampering | malformed CBOE payload | mitigate | CboeResponseSchema.safeParse at adapter entry → Result.err, never throws to transport (RESEARCH Pattern 1) |
| T-02-08 | Denial of Service | unbounded chain write (~20k contracts) | mitigate | fetchChain filter (≤90 DTE, ±10% strike) bounds write volume at use-case layer before persist |
| T-02-09 | Tampering | SQL injection via contract symbols | mitigate | Drizzle parameterized insert/onConflictDoNothing only; no raw template interpolation |
| T-02-10 | Information Disclosure | raw error/stack in fetch failures | mitigate | Errors mapped to {kind,message} Result; adapters log field-level only, no payload dumps |
</threat_model>
