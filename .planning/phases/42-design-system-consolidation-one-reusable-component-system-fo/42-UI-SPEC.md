# Phase 42 — UI Design Contract

> Visual and interaction contract for the design-system consolidation phase. This phase
> ships zero new screens — it is a **convergence contract**: one `DataTable` primitive
> replaces two hand-rolled tables, one `Button` replaces two implementations, and the

## Spacing Scale

Declared values (must be multiples of 4) — unchanged project-wide scale, confirmed still
governing this phase's work:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, `gap-1` |
| sm | 8px | `px-2` cell horizontal padding, `gap-2` |
| md | 16px | Panel padding (`p-3`≈12px is the one established sub-scale exception below), default element spacing |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

**Exceptions (existing, locked — do not invent new ones):**

- `Panel` padding is `p-3` (12px), not 16px — established in `components/system/index.tsx`,
  predates this phase, out of scope to change.
- **DataTable cell padding is the CONVERGENCE TARGET, not the 8pt scale.** CandidateTable's
  current chrome (`px-2 py-1.5` → 8px horizontal / 6px vertical) is the canonical dense
  terminal density per CONTEXT's "Chrome reference = CandidateTable." Overview's
  PositionsTable currently uses `px-2 py-1` (8px / 4px) — it converges UP to `py-1.5` when
  it migrates onto DataTable. This is the one intentional density delta the migration
  produces; it is what makes the two tables "indistinguishable" per the user's complaint.
- Header cell padding: `px-2 py-1.5` (8px/6px), identical to body cells — no extra header
  padding beyond what CandidateTable already ships.

## Typography

Two-weight, existing-scale contract (unchanged — this phase documents, does not alter):

| Role | Size | Weight | Line Height | Where |
|------|------|--------|-------------|-------|
| Table header label | 10px | 600 (semibold, `font-display`) | default (single line) | `SectionLabel`-style: `tracking-[0.09em] uppercase text-dim` |
| Table body / data cell | 11px | 400 (regular, `font-mono`) | default, `tabular-nums` | Every numeric/text `<td>` |
| Table secondary line | 9-10px | 400 (`font-mono`), `text-dim`/`text-muted-foreground` | default | Expiry sub-line, badge text (`PASTED`, event chip) |
| Panel/section heading | 10px | 600 (`font-display`) | default | `SectionLabel` (unchanged, reused by DataTable's own section wrapper if any) |

Body-text weight = 400. Heading/label weight = 600. No third weight is introduced by this
phase — `font-bold` seen on score cells (`font-bold`) is an intra-cell emphasis utility, not
a new declared weight tier; it stays exactly where CandidateTable already uses it (score
value only).

Font-family split is locked: `font-mono` (JetBrains Mono) for every data/numeric surface
(tabular alignment matters for a trading table), `font-display` (Space Grotesk) only for
uppercase tracked labels/headers. DataTable's `<table>` root sets `font-mono text-[11px]
tabular-nums` exactly as CandidateTable does today; header `<th>` elements set
`font-display text-[10px] font-semibold tracking-[0.09em] uppercase text-dim`.

## Component Contract — `DataTable` Primitive

New file: `apps/web/src/components/system/DataTable.tsx` (system/ layer — molecules, per
`components/system/index.tsx`'s own doc comment: "Atoms → ui/*, Molecules → system/*").

**Column-def shape** (generic over row type `T`):

```ts
interface DataTableColumn<T> {
  readonly key: string;                    // stable key + default aria-sort id
  readonly header: React.ReactNode;
  readonly align?: "left" | "right";       // default "right" (numeric-table convention)
  readonly mono?: boolean;                 // defaults true at the <table> root; per-column override rare
  readonly sortable?: boolean;
  readonly width?: string;                 // literal Tailwind width class, not px math
  readonly render: (row: T) => React.ReactNode;
}
```

**Behavior DataTable owns** (per CONTEXT decisions, generalized from CandidateTable):

- Sticky header (`sticky top-0 z-10 bg-panel`) — the header must stay opaque over
  scrolling body rows, exact class reused verbatim from CandidateTable.
- Sort cycling: click an unsorted sortable column → desc; click the active column → flips
  asc/desc; `aria-sort="ascending"|"descending"|"none"` set correctly per column. Reuses
  `cycleSort`'s exact state-machine (new column always enters at desc).
- Row hover (`hover:bg-line/40`) and selected-row highlight (`border-l-2 border-l-violet
  bg-violet/[0.06]`) — both exactly CandidateTable's current classes; PositionsTable's
  highlighted-row treatment (`bg-raise/20`, no left-border) is a DIFFERENT visual and is
  NOT collapsed into this — DataTable exposes both as caller-supplied row `className`
  (a `rowClassName(row)` fn), not a hardcoded "selected" boolean prop, so each caller keeps
  its own selection semantics (CandidateTable = border+tint, Overview = flat highlight).
- `onRowClick(row)` — row-level click handler; per-cell click-stop (e.g. the Combine/×
  action cell, Overview's checkbox cell) stays the CALLER's `render()` responsibility via
  its own `onClick={(e) => e.stopPropagation()}`, not a DataTable feature.
- Internal-scroll wrapper: `wrapperClassName` passthrough on the outer `<div>` (no-scroll
  layout law — the page never scrolls, the table's own box does). Desktop rail:
  `min-h-0 flex-1 overflow-y-auto`. Mobile: `-mx-4 max-h-[…] overflow-x-auto overflow-y-auto
  px-4` (horizontal scroll explicitly OK on mobile, user-locked 2026-07-14).
- `tableClassName` passthrough on the `<table>` itself (mobile sets a `min-w` so columns
  never crush — existing CandidateTable prop, preserved verbatim).
- `rowTestId(row)` fn — REQUIRED prop, no default. Existing testids MUST survive:
  `candidate-row-{id}` (CandidateTable → `rowTestId={(c) => \`candidate-row-${c.id}\`}`) and
  `position-row-{key}` (PositionsTable → `rowTestId={(r) => \`position-row-${r.key}\`}`).
  These are the literal row-level testids referenced by the phase's locked constraints —
  there is no single "positions-table" or "candidate-table" container testid today to
  preserve; only the per-row ones are load-bearing for existing tests.
- Trailing per-row actions slot: exposed via the LAST column's `render()` (no separate
  `actions` prop) — matches CandidateTable's existing Combine/Remove cell, which is just
  the final column today. Do not add a bespoke actions API the two current tables don't
  need.
- Expandable detail row (Overview's verdict-detail `<tr>` under a selected position): NOT
  a DataTable feature this phase — CONTEXT's migration order keeps PositionsTable's
  `expandedRowKey`/`VerdictDetailBody` `<Fragment>` pattern as caller-owned JSX rendered
  AFTER the mapped `<DataTable>` rows are unavailable to interleave generically; the
  planner/executor should decide whether DataTable needs a `renderRowDetail?(row)` escape
  hatch or whether PositionsTable keeps its own `<tbody>`-level composition and only the
  `<thead>`/base row markup route through DataTable. **Flagged as an open implementation
  choice, not resolved here** — CONTEXT does not lock it and either approach satisfies "one
  DataTable, converged chrome."
- Net/total row (Overview's `border-t border-line font-semibold` summary `<tr>`): same
  status as the expandable detail row — caller-composed, not a DataTable-owned feature.

**What DataTable does NOT own:** empty-state copy (callers early-return before mounting),
column-level filtering/pagination (neither table has it, YAGNI), row selection state
management (callers own `selectedId`/`highlightedRowKey` and pass derived `rowClassName`).

## Component Contract — Button Consolidation

- `components/system/Button.tsx` is the SOLE button component after this phase. It already
  has 16 consuming files; it is richer (`variant` + `tone` + `active` + `size`) than
  `components/ui/button.tsx`.
- Three call sites migrate: `Login.tsx`, `RebuildButton.tsx`, `components/ui/dialog.tsx`
  (internal close buttons). Map each `ui/button` `variant` to the nearest `system/Button`
  equivalent:
  - `variant="default"` (dialog's implicit default, Login's primary submit) → `system/Button`
    `variant="primary"`.
  - `variant="outline"` (dialog footer close) → `system/Button` `variant="secondary"`.
  - `variant="ghost"` (dialog corner ✕) → `system/Button` `variant="ghost"`.
  - `variant="destructive"` (none currently used at these 3 sites, but the mapping is
    `system/Button` `variant="destructive"` if one appears during migration).
  - `size="icon-sm"` (dialog corner ✕) → nearest `system/Button` `size` is `"xs"`
    (`system/Button` has no icon-only size tier — `xs`'s `px-[7px] py-0.5` is the closest;
    visual drift on the dialog ✕ button is EXPECTED and ACCEPTABLE per CONTEXT's "Visual
    drift on those 3 surfaces is acceptable and desired").
- **Known integration risk (flag for planner, not resolved here):** `ui/dialog.tsx` uses
  base-ui's `render={<Button variant="outline" />}` prop-merge pattern (`DialogPrimitive
  .Close render={...}`), which clones the given element and merges base-ui's own
  a11y/event props onto it. `system/Button` spreads `...props` onto a native `<button>` and
  forwards no `ref`, so the `render` prop-merge should still work mechanically (spread
  covers arbitrary extra props) but has NOT been verified against base-ui's ref-forwarding
  requirement in this research pass — the executor must smoke-test both `DialogPrimitive
  .Close render={<Button .../>}` sites after the swap, not just visually eyeball them.
- Delete `components/ui/button.tsx` only after all 3 call sites are migrated and green.
- No new `ButtonVariant`/`ButtonTone`/`ButtonSize` values are needed — the existing enums
  already cover every mapped case above.
