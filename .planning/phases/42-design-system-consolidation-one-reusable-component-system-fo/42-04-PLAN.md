<objective>
Fold the three `components/ui/button` call sites into `components/system/Button` and delete
the duplicate, and sweep the two inline Panel-gradient duplicates that live on the same files
(RESEARCH Pitfall 2 — the token sweep and the Button migration land together on Login.tsx and
RebuildButton.tsx because they are the same files already open). Migration Manifest step 4 + the
Login/RebuildButton half of step 5.

Variant mapping (UI-SPEC "Component Contract — Button Consolidation"):
- ui `variant="default"` (Login submit) → system `variant="primary"`
- ui `variant="outline"` (dialog footer Close, RebuildButton Cancel, Rebuild trigger) → system `variant="secondary"`
- ui `variant="ghost"` (dialog corner ✕) → system `variant="ghost"` (name unchanged)
- ui `variant="destructive"` (RebuildButton confirm) → system `variant="destructive"`
- ui `size="icon-sm"` (dialog ✕) → system `size="xs"` (no icon-only tier; visual drift EXPECTED + acceptable per CONTEXT)
- ui `size="sm"` → system `size="sm"`

Known integration risk (UI-SPEC + RESEARCH A1, unresolved): `system/Button` is a plain
function component with NO `React.forwardRef` (Button.tsx:88-100). base-ui's
`DialogPrimitive.Close render={<Button/>}` clones the element and merges its own a11y/event
props (possibly a ref) onto it. The spread-props pattern SHOULD cover it, but focus-return is
unverified — mandatory smoke test after the swap; add `forwardRef` only if the smoke test fails.

Purpose: one Button, one token source for these surfaces.
Output: 3 files migrated, ui/button.tsx deleted, Login/RebuildButton gradient dup swept.
</objective>
