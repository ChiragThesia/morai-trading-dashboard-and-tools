<objective>
Migrate Overview's `PositionsTable` onto `DataTable<PositionRow>` — Migration Manifest step 3,
the second and harder table. This is what makes the Overview positions table and the Analyzer
candidate table indistinguishable (the user's original complaint): PositionsTable's header
converges to the sticky `bg-panel` header and its cells converge UP from `px-2 py-1` to
`px-2 py-1.5` (UI-SPEC "the one intentional density delta").

RESOLVED OWNERSHIP DECISION (locking UI-SPEC Pitfall 5 / RESEARCH assumption A2 now, per the
phase constraint "PLANNER MUST DECIDE ownership now"): **DataTable owns all header + base-row
chrome; the two extra row kinds are caller-composed through the slots Plan 01 built** —
- the expandable verdict-detail `<tr>` (Overview.tsx:306-313) is passed via
  `renderRowDetail={(r) => expanded ? <detailTds/> : null}`, and
- the Net-total `<tr>` (Overview.tsx:317-368) is passed via `footer={<NetTotalRow/>}`.
This keeps ONE DataTable rendering every header cell and base row (full convergence, zero
hand-duplicated `<th>`/`<tr>` chrome) while the caller retains the detail/total markup it
alone knows how to build. The alternative (PositionsTable hand-composing its own `<tbody>`)
was rejected because it would re-duplicate the header/row chrome the whole phase exists to
delete. DataTable grows NO expansion/total STATE — the caller keeps `expandedRowKey`/`excluded`.

Purpose: one converged table across Overview + Analyzer.
Output: PositionsTable rewritten onto DataTable; Overview.test.tsx green, unchanged.
</objective>
