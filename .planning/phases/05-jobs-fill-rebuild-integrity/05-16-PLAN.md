<objective>
Add the locked-decision fast-check property tests over randomized fill/roll/partial-fill
sequences feeding sync-fills + rebuild. Round 2's example tests cover specific cases; these
properties harden the invariants across the whole input space so a future regression on the
WR-A1/WR-A2/WR-A3 fixes is caught.

Locked properties (05-GAPS-2.md "Property tests"):
1. **No double-count:** no fill is counted in more than one event (sum of event qty ==
   sum of distinct paired fills; orphaned fills excluded).
2. **Idempotent sync:** re-running sync over the same fills is a no-op — the event count is
   stable across a second sync (relies on WR-A2 processed-tracking).
3. **Rebuild reconciliation:** after rebuild, openNetDebit/closeNetCredit equal the summed
   events for OPEN/CLOSE and ROLL calendars (relies on WR-A1 eventType recompute).
4. **No id collision:** distinct (activityId, legIndex) keys → distinct fill UUIDs
   (relies on WR-A3 total-nibble mapping).

Purpose: lock the round-2 correctness invariants against random inputs, per the user's
"full round-2 fix … plus fast-check property tests" decision.
Output: two property-test files exercising the real use-cases + the in-memory twins (no
Docker), each with numRuns high enough to be meaningful (≥ 200; ≥ 1000 for the pure
collision property, matching the iv-inversion suite convention).
</objective>

      (via makeSyncFillsUseCase) with capturing/mutating fake ports + a shared fill store, and
      the in-memory fills twin for recompute.
    - Arbitraries (fast-check):
      - A calendar with two legs (front/back OCC symbols) and a status.
      - An array of RawFills over those two legs with random side/qty(>0)/price(>0)/orderId,
        including: multiple partial fills sharing (orderId, leg) (partial-fill buckets), and
        CLOSE+OPEN pairs on the same orderId across the two legs (ROLL candidates). Constrain
        generators so qty>0 and prices are finite positive (avoid the aggregation-error branch
        unless explicitly testing it).
    - Properties (fc.assert, numRuns ≥ 200):
      - **P1 no double-count:** after one sync, sum over emitted events of (the count of fill
        ids composing each event) == the number of DISTINCT fill ids that were paired (i.e. not
        orphaned). No fill id appears in two events. Use the capturing storeCalendarEvent +
        markFillsProcessed to reconstruct which fills went into which event.
      - **P2 idempotent:** run sync twice over the SAME fill store (markFillsProcessed mutates
        the store so the second readUnprocessedFills returns only unmarked fills). Assert the
        event count after the second sync equals the count after the first (no new events,
        no duplicates).
      - **P2b partial-fill growth:** start with bucket {A}, sync; add {B} to the same
        (orderId, leg) and sync again. Assert exactly ONE additional event covering only {B},
        and the {A} event is unchanged — no fill double-counted (the WR-A2 invariant).
      - **P3 rebuild reconciliation:** after seeding the emitted events into the twin and
        running recompute (or running rebuild end-to-end), openNetDebit == sum of OPEN
        netAmounts + sum of ROLL open-leg debits, and closeNetCredit == sum of |CLOSE
        netAmounts| + sum of ROLL close-leg credits (the WR-A1 invariant), within a float
        tolerance (toBeCloseTo, 5 digits).
    - Run; all properties pass. If a property finds a counterexample, treat it as a real bug in
      05-14/05-15 and fix the source (do not shrink the generator to hide it).
  </behavior>
