<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker → Schwab OAuth refresh endpoint | The handler triggers token refresh; refresh tokens + access tokens cross this boundary and are persisted encrypted (pgcrypto) by the existing broker-tokens repo |
| worker → broker_tokens.last_refresh_error → server /api/status surface | Per-app failure state is persisted on the existing broker_tokens row (no new table) and read back by getStatus across the process boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-11 | Information Disclosure | token values in logs / in last_refresh_error | mitigate | Use-case + handler log ONLY appId + timestamps; the persisted last_refresh_error stores the error string (appId + reason) only — never accessToken/refreshToken/TOKEN_ENCRYPTION_KEY (workflow.md credential rule); acceptance criteria greps for token field logging |
| T-05-12 | Denial of Service | one app's failure blocking the other | mitigate | Promise.allSettled guarantees both apps attempted regardless of the other's outcome (D-13); test asserts both refresh fns invoked when one rejects |
| T-05-13 | Availability | silent refresh-token hard expiry | mitigate | isNearExpiry surfaces a warning at day 6 of 7 (D-14) so the operator re-auths before a data gap; refreshIssuedAt never reset on refresh (Pitfall 3) |
| T-05-14 | Tampering | encryption key in SQL | mitigate | recordRefreshOutcome UPDATEs last_refresh_error with bound `$N` params only; the pgcrypto key still appears solely as a bound param in the existing token write path — this plan adds no sql.raw touching the key |
</threat_model>
