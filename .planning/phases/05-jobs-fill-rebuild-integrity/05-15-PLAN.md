<objective>
Close the two correctness warnings that require a data-model change, plus the twin-fidelity
warning they depend on:

- **WR-A2 (JRNL-01):** "processed" is currently defined as "id present in orphan_fills". A
  PAIRED fill is written only as a `fill_ids_hash` (a hash, not per-fill ids) and is never
  added to orphan_fills, so it stays "unprocessed" forever — every sync re-reads and
  re-pairs the full lifetime fills table (unbounded), and partial-fill growth breaks
  idempotency: a bucket {A} at run N and {A,B} at run N+1 emits a SECOND event for the larger
  aggregate while the run-N event for {A} still exists, double-counting the leg. Fix: track
  processed fills explicitly with a `fills.processed_at` column + a `ForMarkingFillsProcessed`
  port; syncFills marks a bucket's fills processed once its event is stored (and orphan-parked
  fills are also processed); `readUnprocessedFills` = `WHERE processed_at IS NULL AND id NOT IN
  orphan_fills`. Semantics (documented): each fill lands in exactly ONE event; later fills for
  the same order/leg form a NEW event covering only the new fills — no fill is counted twice.

- **WR-A1 (JRNL-01):** a ROLL event stores a combined `netAmount = openDebit − closeCredit`,
  and `recomputeCalendarAmounts` buckets by SIGN, so a roll collapses into one aggregate and
  SC5 reconciliation fails for any calendar containing a roll. Fix: recompute sums by
  `eventType` — OPEN.netAmount → openNetDebit, CLOSE.netAmount → closeNetCredit, ROLL splits
  its open-leg debit → openNetDebit and close-leg credit → closeNetCredit. Persist the ROLL's
  open/close components EXPLICITLY (dedicated columns the recompute reads), not by re-parsing
  the free-JSON legBreakdown.

- **WR-A4 (JRNL-01):** the in-memory `seedEvent` keeps only `{calendarId, netAmount}` and
  cannot represent a ROLL, so once WR-A1 lands the twin diverges from Postgres and the shared
  contract suite stops exercising equivalent behavior. Fix: mirror the full `SeedEvent` shape
  (eventType, legOccSymbol, rolledFromOccSymbol, fillIdsHash, and the ROLL open/close
  components) in the twin and store it.

Purpose: make re-sync provably idempotent (no double-count, no unbounded re-pair) and make
rebuild reconciliation (SC5) hold for calendars that contain a ROLL — both on the trade
ledger.
Output: a migration + the in-memory twin updated in the SAME plan (migration discipline), a
new port, a recompute that is eventType-aware with explicit ROLL components, and a twin that
can model every event the Postgres adapter does.
</objective>
