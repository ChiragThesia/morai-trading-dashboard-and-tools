<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client/Claude Code → POST /api/jobs/:name/trigger | Untrusted caller requests a job enqueue; job name + calendarId are user-supplied and must be validated + authz-checked |
| MCP client → trigger_job tool | Same untrusted surface via the MCP transport |
| rebuild-journal → calendar_events (Postgres) | Delete-then-reinsert mutates the trade ledger; scoped to one calendarId |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-21 | Elevation of Privilege | unauthenticated trigger_job | mitigate | The HTTP route is mounted inside the existing bearer-token middleware; the MCP transport is already auth-guarded — acceptance criteria asserts the route is not exposed unauthenticated (Security Domain) |
| T-05-22 | Input Validation | malicious :name / calendarId | mitigate | z.enum(TRIGGERABLE_JOBS) rejects arbitrary job names; triggerJobPayload validates calendarId as a UUID; safeParse at the MCP boundary (V5 Input Validation) |
| T-05-23 | Tampering / Repudiation | rebuild data loss | mitigate | Delete-then-reinsert is scoped to one calendarId and is purely derived from fills (fills are the source of truth, JRNL-01); idempotency + SC5 reconciliation prove no data is lost or duplicated |
| T-05-24 | Denial of Service | trigger flooding / duplicate enqueue | mitigate | enqueueJob uses the dedupe key (rebuildDedupeKey:{calendarId}) so repeated triggers within a window collapse to one job via pg-boss singletonKey (05-04) |
</threat_model>
