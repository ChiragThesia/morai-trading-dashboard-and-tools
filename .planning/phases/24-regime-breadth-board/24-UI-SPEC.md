## Spacing Scale

This phase adds zero new spacing values — it composes the **already-shipped** `Panel`,
`MetricChip`, and `MacroCard`'s `Tile` molecules exactly as they exist today. The
effective scale those molecules already use (verified in `system/index.tsx` and
`MacroCard.tsx`) is documented here rather than the generic 8-point default, since
CONTEXT.md locks "MetricChip visual language" — reusing the shipped padding/gap values
verbatim is the contract, not a new design decision.

| Token | Value | Usage (existing, reused) |
|-------|-------|---------------------------|
| xs | 4px | `gap-1` between a band-status dot and its label (new, matches `MetricChip`'s existing `gap-1.5` family) |
| sm-half | 6px | `gap-1.5` (MetricChip internal gap), `py-1.5` (chip/tile vertical padding) — Tailwind's half-step, already load-bearing in every shipped molecule |
| sm | 8px | `gap-2` (section header row gaps), `px-2` (Tile padding) |
| md | 12px | `px-3` (MetricChip horizontal padding), `p-3` (Panel padding) |
| lg | 16px | Panel-to-panel grid gaps at `gap-3`→`gap-4` breakpoints (existing `Overview.tsx` section spacing) |

Exceptions: the 6px (`sm-half`) step is not a multiple of 4 but is the existing,
shipped value in `MetricChip`/`Tile` — reusing it exactly is required by the "MetricChip
visual language" lock, not a new exception introduced by this phase.

## Typography

This system collapses "Label" and "Heading" into one visual tier (`SectionLabel`/
`PanelHeading` — there is no separate larger heading size anywhere in the shipped UI).
Declaring both rows below to the same value is accurate to the existing system, not a
contract gap.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (provenance prose, mirrors `WhyPanel`'s sentence style) | 12px (`text-xs`) | 400 (regular, no weight class) | 1.45 (`leading-[1.45]`) |
| Label (chip label, `MetricChip`/`Tile` uppercase caption) | 10px (`text-[10px]`) | 600 (`font-semibold`) | 1.2 (default, tracked `0.06–0.09em`) |
| Heading (`PanelHeading`/`SectionLabel` — same tier as Label in this system) | 10px (`text-[10px]`) | 600 (`font-semibold`) | 1.2 |
| Display (indicator current value, `MetricChip`'s value slot) | 16px (`text-base`) | 700 (`font-bold`) | 1.2 (tabular-nums, no override) |

Weight note: the system uses 3 weights total in practice (400 prose / 600 label / 700
display bold) — not the generic 2-weight default. This is the existing, shipped
convention (`MetricChip` label is `font-semibold`, its value is `font-bold`); the board
does not introduce a 4th weight.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-panel` → `bg-panel2` (`#0f1521` → `#0c111a`, the `Panel` gradient) | Page background, every card surface |
| Secondary (30%) | `bg-raise` (`#161d2b`), used at `/40` opacity for tiles/chips | Board chip fill, `Tile`/`MetricChip` background |
| Accent (10%) | `text-violet` (`#a78bfa`) | Reserved for: focus/selection ring (`--color-ring`), the payoff chart's T+0 curve, and any future interactive/selected chip state — **never** used for band color on this board (bands use the semantic up/amber/down triad below, per CONTEXT.md's banding lock) |
| Destructive | `text-down` / `bg-downd` (`#ef5350` / `#3e1f23`) | Crisis band only — same token `MetricChip`'s `alert` prop already uses for negative-gamma/loss emphasis |

Accent reserved for: focus rings, the payoff-chart T+0 curve color, and future
click-to-select chip state. Never band color.

### Band color triad (domain-specific — additive to the 60/30/10 split)

CONTEXT.md locks band colors to the existing semantic tokens (calm/warning/crisis).
This is not a new palette — it is `MetricChip`'s existing `alert`/sign-color convention
extended to a 3-state triad instead of 2:

| Band | Token | Hex | Existing precedent |
|------|-------|-----|---------------------|
| calm | `text-up` / dot `bg-up` | `#26a69a` | `signClass()`'s positive-value color, used everywhere (P&L, Δ, γ regime "DAMPEN") |
| warning | `text-amber` / dot `bg-amber` | `#f0b429` | γ flip chip, staleness-warning dot (`gexFresh` amber state) |
| crisis | `text-down` / dot `bg-down`, chip background `bg-downd` | `#ef5350` / `#3e1f23` | `MetricChip alert={true}` (negative-gamma "AMPLIFY" regime), `signClass()`'s negative color |

## Component & Layout Contract

*(Additive detail beyond the template — the specific decisions the planner/executor need that don't fit the generic table shape above.)*

### Placement (USER-LOCKED, CONTEXT.md)

New `<section>` on `Overview.tsx`, own `SectionLabel` ("Regime & breadth"), placed
**between** the existing "Positioning & macro detail" section (`CotCard` + `MacroCard`)
and the "Book & system" section — same structural pattern as those two sections (a
`SectionLabel` + a `Panel` or grid of `Panel`s), not inside either existing grid and not
a new route/tab. This keeps `CotCard`/`MacroCard`'s existing 2-column grid untouched
(zero risk of squeezing 4 new chips into an already-tuned 2-col layout) while still
satisfying "Overview tab, existing grid system, not a new page."

### Chip anatomy (per indicator — 4 total: VIX/VIX3M, VVIX, VIX9D/VIX, HY OAS)

One `Panel` titled "Regime & breadth" containing a `grid grid-cols-2 gap-2 md:grid-cols-4`
of chip cells (mirrors `MacroCard`'s `Tile` grid density, scaled to 4 not 5/3 columns).
Each chip is a `Tile`-shaped cell (not `MetricChip`'s pill shape — `Tile`'s rectangular
`rounded-sm bg-raise/40 px-2 py-1.5` reads better at 4-up density than a pill row) with:

1. **Row 1 — label + info trigger:** `SectionLabel`-style 10px uppercase label (e.g.
   `"VIX/VIX3M"`) + inline `ⓘ` `Tooltip` trigger (info affordance, no visible text)
   showing `source` + `rationale` on hover, exact reuse of the `PositionsTable` "IV n/a"
   badge+tooltip pattern.
2. **Row 2 — band dot + value:** a `size-1.5` rounded dot colored by band
   (`bg-up`/`bg-amber`/`bg-down`) + the current value at Display tier (16px, bold,
   tabular-nums), colored to match the dot (`text-up`/`text-amber`/`text-down`).
3. **Row 3 — as-of date:** `"as of {date}"` at 10px `text-dim`, date-only (no
   time-of-day) — the MACRO-03 EOD-not-intraday guarantee made visible per chip, not
   just in a single header stamp.

`data-testid` convention (extends the existing `macro-value-{id}` precedent):
`regime-chip-{id}`, `regime-value-{id}`, `regime-band-{id}`, `regime-asof-{id}`,
`regime-why-{id}` — `{id}` is the payload's own `id` field (`"vix-term-structure"`,
`"vvix"`, `"vix9d-vix"`, `"hy-oas"`), not an index.

### States

- **Loading** (`isPending && data === undefined`): single centered `Panel` message, see
  Copywriting Contract → Loading state.
- **Empty** (`data` resolves to `[]`): single centered `Panel` message, see Copywriting
  Contract → Empty state.
- **Partial** (1–3 of 4 indicators present): render only the present chips in the same
  grid — grid auto-flows, no empty placeholder cell for a missing indicator (never a
  fabricated dash chip).
- **Loaded, all 4 present:** the 4-chip grid described above.

### Interaction

- Hover/focus on the `ⓘ` trigger → `Tooltip` shows `source` + `rationale` (BOARD-02).
  Keyboard-focusable (Radix `Tooltip` default), same accessibility floor as the existing
  IV-n/a tooltip precedent — no new a11y work required, no new pattern introduced.
- No click-to-select, no drill-down, no navigation. This is a read-only glance board
  (matches CONTEXT.md's explicit scope: display-only banding this phase, hard-gate
  wiring deferred to Phase 28).
